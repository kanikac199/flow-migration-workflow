# 6.1 ProcessTracker Architecture

> **Phase 3 — Async Workflow Migration**
>
> **Prerequisites:**
> - Phase 1 and Phase 2 gateway/flow migrations are stable
> - Read [2.2_architecture_comparison.md](../2_overview/2.2_architecture_comparison.md) for the stateless nature of connector-service

---

## Overview

euler_txn's ProcessTracker is a persistent task queue stored in MySQL. It powers all async workflows — transaction status syncs, async refund processing, webhook delivery, mandate execution, tokenization, and auto-void cleanup.

**This is Phase 3 migration** — the highest risk phase because it involves async state management. The recommendation is to keep ProcessTracker in euler_txn for the near term and only migrate the gateway calls within each workflow to connector-service.

---

## Architecture

```
┌──────────────┐     schedule     ┌─────────────────┐     execute     ┌────────────────┐
│ Flow handler  │ ──────────────► │ process_tracker  │ ──────────────► │ Workflow runner │
│ (creates PT)  │                 │ table (MySQL)    │                 │ (picks up task) │
└──────────────┘                  └─────────────────┘                  └────────────────┘
                                         │
                                    Fields:
                                    - trackingData (JSON)
                                    - status (PENDING/RUNNING/COMPLETED)
                                    - retryCount
                                    - nextRetryAt
                                    - runnerName
```

### How It Works

1. **Scheduling:** A flow handler (e.g., after `initiateTxn`) creates a `process_tracker` row in MySQL with:
   - `runnerName` — identifies which workflow to execute (e.g., `TXN_SYNC_WORKFLOW_TRANSIENT`)
   - `trackingData` — JSON blob with all context needed (txn ID, gateway, merchant ID, etc.)
   - `nextRetryAt` — when to first execute
   - `retryCount` — starts at 0
   - `status` — starts as `PENDING`

2. **Polling:** A scheduler thread polls the `process_tracker` table for tasks where:
   - `status = PENDING`
   - `nextRetryAt <= NOW()`
   - `runnerName` matches a registered runner

3. **Execution:** The matched workflow runner:
   - Decodes `trackingData`
   - Runs pre-checks (terminal state? max retries? health check?)
   - Acquires a lock (to prevent duplicate execution)
   - Executes the workflow logic (e.g., call gateway status sync API)
   - Updates the task: either complete it or reschedule with incremented `retryCount`

4. **Retry Logic:** Failed tasks get rescheduled with exponential backoff:
   ```
   nextRetryAt = NOW() + baseDelay * 2^retryCount
   ```
   With jitter and gateway-specific retry policies.

## connector-service: No Scheduler

connector-service is **stateless** — it has no database, no task queue, and no scheduler. It processes requests synchronously.

This means async workflows **cannot** move to connector-service as-is. The options are:

### Option 1: Keep ProcessTracker in euler_txn (Recommended for Now)

- ProcessTracker stays in euler_txn
- The gateway API call within each workflow is delegated to connector-service via bridge
- No architectural changes to async flow management

```
euler_txn ProcessTracker
    |
    v
Workflow runner
    |
    v
shouldUseConnectorService?
    |
    YES --> bridge call to connector-service (e.g., PSync flow)
    |
    NO  --> direct gateway call (old path)
```

**Pros:** Minimal risk, uses proven bridge pattern
**Cons:** euler_txn must stay running for async workflows

### Option 2: New Scheduler Service (Future)

Build a separate Rust-based scheduler service that:
- Has its own database for task persistence
- Implements retry logic, exponential backoff, jitter
- Calls connector-service for gateway interactions
- Exposes an API for scheduling tasks

**Pros:** Full migration away from euler_txn
**Cons:** Significant new service to build and operate

### Option 3: External Queue (Future)

Migrate to an external queue system:
- **Kafka consumers** — scheduled via delay topics
- **Temporal** — workflow orchestration engine
- **SQS + Lambda** — serverless task processing

**Pros:** Leverages proven infrastructure
**Cons:** Different operational model, migration complexity

## Migration Approach (Near Term)

For Phase 3, the recommended approach is **Option 1** — keep ProcessTracker, migrate gateway calls:

1. **Identify gateway calls** within each workflow runner
2. **Replace direct gateway calls** with bridge calls to connector-service
3. **Feature-flag the bridge call** within the workflow (same pattern as Phase 1)
4. **Test thoroughly** — async workflows have timing-dependent bugs that are hard to catch
5. **Monitor retry behavior** — ensure connector-service errors trigger correct retry logic

### What Changes in Each Workflow

| Workflow Component | Changes? | Detail |
|-------------------|----------|--------|
| Task scheduling | No | Same `process_tracker` insert |
| Task polling | No | Same scheduler polling |
| Pre-checks (terminal state, retries) | No | Same logic in euler_txn |
| Lock acquisition | No | Same distributed lock |
| **Gateway API call** | **Yes** | Replaced with connector-service bridge call |
| Response handling | Minimal | Map connector-service response to existing format |
| Retry/complete logic | No | Same retry scheduling |
