## Overview

The TxnSync workflow is the **most critical async workflow** in euler_txn. It polls gateways for transaction status updates when the initial payment response is non-terminal (pending, redirecting, timeout). Without TxnSync, transactions can get stuck in intermediate states indefinitely.

**Source:** `oltp/src/Engineering/WorkFlow/TxnSyncWorkFlow.hs` (461 lines)

**Runner keys:** `TXN_SYNC_WORKFLOW_TRANSIENT` (short-lived), `TXN_SYNC_WORKFLOW_AGGRESSIVE` (higher frequency)

---

## Logic Flow

```
1. Decode trackingData → TxnProcessTrackingData
    |
    v
2. Check multi-cell migration applicability
    |
    v
3. Pre-checks:
    |
    ├── a. Gateway found?              → If not, fail task
    ├── b. Terminal state reached?      → If yes, complete task (no-op)
    ├── c. Max retry exceeded?          → If yes, mark as failed/timeout
    └── d. Combination blocked?         → If yes (gateway + merchant blocked), skip
    |
    v
4. OL Health check
    |
    ├── Health DOWN → defer (exponential backoff), reschedule
    └── Health UP   → continue
    |
    v
5. Acquire lock (retry up to 3x with jitter)
    |
    ├── Lock failed → reschedule with short delay
    └── Lock acquired → continue
    |
    v
6. Call initSyncGatewayStatus (→ gateway sync API)
    |
    v
7. Check result:
    |
    ├── Terminal state (CHARGED/FAILED/VOIDED)
    │   → Update TxnDetail status
    │   → Complete task
    │   → Fire merchant webhook (if applicable)
    |
    ├── GSM intermediate terminal (gateway-specific mapping says "stop retrying")
    │   → Special retry schedule (longer delays)
    │   → May transition to AGGRESSIVE runner
    |
    └── Still pending
        → Increment retryCount
        → Calculate next retry time (exponential backoff)
        → Reschedule task
```

## Key Concepts

### TxnProcessTrackingData

The JSON blob stored in `process_tracker.trackingData`:

```haskell
data TxnProcessTrackingData = TxnProcessTrackingData
  { txnId        :: Text        -- Transaction ID
  , orderId      :: Text        -- Order ID
  , merchantId   :: Text        -- Merchant ID
  , gateway      :: Gateway     -- Which gateway to sync with
  , txnUuid      :: Text        -- Unique transaction UUID
  , maxRetries   :: Int         -- Max retries before timeout
  , retryDelay   :: Int         -- Base delay between retries (seconds)
  }
```

### GSM (Gateway Status Mapping)

GSM is a configuration that maps gateway-specific response codes to internal actions:

| GSM Action | Meaning | TxnSync Behavior |
|-----------|---------|-----------------|
| `TERMINAL` | Gateway says final state | Complete task |
| `RETRY` | Gateway says try again | Reschedule with standard delay |
| `INTERMEDIATE_TERMINAL` | Special case (e.g., "awaiting 3DS") | Reschedule with longer delay |
| `BLOCK` | Don't retry this combination | Skip, mark task accordingly |

### Retry Schedule

```
Transient runner:
  Retry 1:  30 seconds
  Retry 2:  60 seconds
  Retry 3:  120 seconds
  Retry 4:  240 seconds
  ...
  Max: ~20 retries over ~24 hours

Aggressive runner:
  Retry 1:  10 seconds
  Retry 2:  20 seconds
  Retry 3:  30 seconds
  ...
  Max: ~10 retries over ~30 minutes
  Then transitions to Transient runner
```

### OL Health Check

Before making the gateway call, TxnSync checks the "OL" (Online/Offline) health status of the gateway. If the gateway is marked as unhealthy:
- The sync is deferred (not attempted)
- The task is rescheduled with exponential backoff
- This prevents hammering a down gateway with sync requests

## Migration: Gateway Call to connector-service

The only part that changes in Phase 3 migration is step 6 — the gateway sync API call:

### Before (Direct Gateway Call)

```haskell
-- In TxnSyncWorkFlow.hs:
initSyncGatewayStatus gateway txnDetail mga = do
  -- Calls Gateway/<Name>/Flow.hs syncTxnStatus function directly
  result <- Gateway.syncTxnStatus gateway txnDetail mga
  return result
```

### After (Bridge to connector-service)

```haskell
-- In TxnSyncWorkFlow.hs:
initSyncGatewayStatus gateway txnDetail mga = do
  useCS <- shouldUseConnectorService gateway merchantId
  if useCS
    then do
      -- Build PSync request for connector-service
      let psyncReq = buildPSyncBridgeRequest txnDetail mga
      csResp <- ConnectorService.Flow.makePSyncRequest psyncReq
      convertPSyncResponse csResp
    else do
      -- Old path: direct gateway call
      Gateway.syncTxnStatus gateway txnDetail mga
```

### connector-service PSync Flow

```rust
ConnectorIntegrationV2<
    connector_flow::PSync,          // Payment Sync marker
    PaymentFlowData,
    PaymentsSyncData,               // Contains connector_transaction_id
    PaymentsResponseData,           // Returns current status
>
```

The PSync flow in connector-service:
1. Builds a status inquiry request to the gateway
2. Calls the gateway API
3. Parses the response
4. Returns the current `AttemptStatus`

## Critical Considerations

### 1. Error Handling Must Not Break Retry Logic

If the connector-service call fails (timeout, 5xx, parse error), the TxnSync workflow must:
- **NOT** mark the task as completed
- **NOT** change the transaction status
- Treat it as a retriable failure and reschedule

```haskell
-- Ensure connector-service errors are treated as retryable:
case connectorServiceResult of
  Right resp -> handleSyncResponse resp  -- Process normally
  Left err   -> do
    logError "Connector service call failed" err
    rescheduleTask (retryCount + 1)      -- Retry, don't fail
```

### 2. Status Mapping Must Be Exact

The status returned by connector-service's PSync flow must map to the same `TxnStatus` that euler_txn's direct gateway call would produce. Any mismatch means:
- Transactions could get stuck in wrong states
- Merchant notifications could fire incorrectly
- Refund eligibility could be wrong

Use the mapping table in [7.1_type_mapping.md](../7_reference/7.1_type_mapping.md) to verify.

### 3. Lock Handling

The distributed lock acquired in step 5 must be released regardless of whether the connector-service call succeeds or fails. Use a `finally` / `bracket` pattern:

```haskell
bracket
  (acquireLock txnId)       -- acquire
  (releaseLock txnId)       -- release (always)
  (\_ -> callPSync ...)     -- action
```

### 4. Timing Sensitivity

TxnSync workflows are timing-sensitive. Adding a connector-service hop adds latency:
- Direct gateway call: ~200-500ms
- Via connector-service: ~300-700ms (additional ~100-200ms for bridge overhead)

This is acceptable because TxnSync is async and not user-facing. But monitor p99 latency to ensure it doesn't cause retry scheduling issues.

## Testing TxnSync Migration

1. **Unit test:** Mock connector-service PSync response, verify TxnSync workflow handles it correctly
2. **Integration test:** With a sandbox gateway, trigger a pending payment and verify TxnSync picks it up and resolves it via connector-service
3. **Shadow test:** Run both old and new paths, compare final transaction states
4. **Retry test:** Simulate connector-service failure, verify retry logic works correctly
5. **Timeout test:** Ensure max-retry timeout behavior is unchanged
