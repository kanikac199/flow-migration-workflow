# 7.1 Type Mapping Reference

> **Reference — Use during implementation**
>
> This document provides complete type mapping tables between euler_txn (Haskell)
> and connector-service (Rust). Consult this whenever you need to translate a
> type from one system to the other.

---

## 7.1.1 Transaction Status (TxnStatus -> AttemptStatus)

This is the **most critical** mapping. Getting this wrong means transactions end up in incorrect states.

| euler_txn `TxnStatus` | connector-service `AttemptStatus` | Notes |
|-----------------------|----------------------------------|-------|
| `STARTED` | `Started` | Initial state |
| `PENDING_VBV` | `AuthenticationPending` | 3DS redirect pending |
| `VBV_SUCCESSFUL` | `AuthenticationSuccessful` | 3DS completed |
| `AUTHORIZING` | `Pending` | Awaiting gateway |
| `AUTHORIZED` | `Authorized` | Pre-auth successful |
| `CHARGED` | `Charged` | Payment captured |
| `PARTIAL_CHARGED` | `PartialCharged` | Partial capture |
| `AUTHENTICATION_FAILED` | `AuthenticationFailed` | 3DS failed |
| `AUTHORIZATION_FAILED` | `Failure` | Payment rejected |
| `JUSPAY_DECLINED` | `Failure` | Pre-gateway rejection |
| `VOIDED` | `Voided` | Pre-auth voided |
| `VOID_INITIATED` | `VoidInitiated` | Void in progress |
| `VOID_FAILED` | `VoidFailed` | Void failed |
| `CAPTURE_INITIATED` | `CaptureInitiated` | Capture in progress |
| `CAPTURE_FAILED` | `CaptureFailed` | Capture failed |
| `AUTO_REFUNDED` | `AutoRefunded` | System-initiated refund |
| `PENDING` | `Pending` | Generic pending |
| `FAILURE` | `Failure` | Generic failure |
| `COD_INITIATED` | *(no direct mapping)* | Cash-on-delivery; handle in bridge |
| `NOP` | *(no direct mapping)* | No operation; handle in bridge |
| `TO_BE_CHARGED` | `Authorized` | Pre-auth awaiting capture |
| `MERCHANT_VOIDED` | `Voided` | Merchant-initiated void |
| `AUTO_VOIDED` | `Voided` | System-initiated void |
| `COMPLETED` | `Charged` | Terminal success |

### Unmapped Statuses

These euler_txn statuses have no direct connector-service equivalent. Handle them in the bridge layer:

- **`COD_INITIATED`** — Cash-on-delivery is not a gateway flow; keep entirely in euler_txn
- **`NOP`** — No-operation status; should never reach connector-service

## 7.1.6 Auth Type Mapping

| euler_txn Auth Pattern | connector-service `ConnectorAuthType` | Example |
|-----------------------|-------------------------------------|---------|
| Single API key in `MerchantGatewayAccount.apiKey` | `HeaderKey { api_key }` | Razorpay |
| API key + secret | `BodyKey { api_key, key1 }` | Paytm |
| API key + secret + merchant ID | `SignatureKey { api_key, key1, api_secret }` | HDFC |
| Certificate-based | `CertificateAuth { certificate, private_key }` | Some banks |
| OAuth (access token flow) | Uses `AccessToken` flow | Stripe Connect |

### How Auth Flows Through the Bridge

```
euler_txn                              connector-service
---------                              -----------------
MerchantGatewayAccount
  .apiKey          --> Header: x-api-key
  .key1            --> Header: x-key1
  .merchantId      --> Header: x-merchant-id
  .authType        --> Header: x-auth (e.g., "HeaderKey", "BodyKey")
                       |
                       v
                   Parsed into ConnectorAuthType
                   by connector-service middleware
```

## 7.1.8 Currency Mapping

euler_txn uses string currency codes (e.g., `"INR"`, `"USD"`). connector-service uses a `Currency` enum.

```rust
// connector-service
pub enum Currency {
    INR,
    USD,
    EUR,
    GBP,
    // ... 50+ currencies
}
```

The bridge must convert:
```haskell
-- euler_txn
currencyStr = fromMaybe "INR" (order ^. _currency)
-- becomes gRPC metadata header:
("x-currency", currencyStr)
```
