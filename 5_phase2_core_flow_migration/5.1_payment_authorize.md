# 5.1 Payment Authorize Flow Migration

> **Phase 2 — Core Transaction Flow Migration**
>
> **Prerequisites:**
> - Phase 1 gateway connector migration is stable for all target gateways
> - Read [2.2_architecture_comparison.md](../2_overview/2.2_architecture_comparison.md) for layer mapping
> - Read [7.1_type_mapping.md](../7_reference/7.1_type_mapping.md) for status/type mappings

## Overview

The Payment Authorize flow is the most complex flow in euler_txn. It handles the full payment initiation lifecycle — from receiving a merchant's pay request to returning a gateway response.

**This is a P0 migration target** but also the highest-complexity one. Phase 2 recommends migrating only the connector-call portion (already done in Phase 1) and keeping orchestration in euler_txn until a dedicated orchestration layer is built.

---

## euler_txn Flow

**Entry point:** `POST /txns` -> `Transaction.decider`
**Source file:** `oltp/src-generated/Product/OLTP/Transaction.hs`

```
HTTP Request
    |
    v
decider()
    |
    v
validateOrder                     -- Check order exists, is valid, not expired
    |
    v
selectGateway                     -- Gateway decider (euler-api-decider service)
    |
    v
createTxnDetail                   -- DB: Insert TxnDetail record
    |
    v
resolvePaymentFlow                -- Determine flow type: card, NB, UPI, redirect, etc.
    |
    v
initiateTxn (gateway adapter)    -- Call Gateway/<Name>/Flow.hs
    |
    v
handleResponse                    -- Parse gateway response, map status
    |
    v
updateTxnStatus                   -- DB: Update TxnDetail with result
    |
    v
scheduleSync (ProcessTracker)     -- Schedule async status sync
    |
    v
return response                   -- PaymentResp to merchant
```

### Key Entanglements

The authorize flow is deeply coupled to:

| Dependency | Location | Impact |
|-----------|----------|--------|
| **Gateway decider logic** | `euler-api-decider` (separate service) | Selects which gateway to route to |
| **200+ feature toggles** | `Config/Constants.hs`, `Config/Options.hs` | Control flow behavior per merchant/gateway |
| **DB reads** | `OrderReference`, `TxnDetail`, `TxnCardInfo`, `MerchantGatewayAccount` | Required context for request building |
| **DB writes** | `TxnDetail` status, `TxnCardInfo`, audit logs | State persistence after gateway response |
| **ProcessTracker** | `WorkFlow/Flow.hs` | Schedules async sync after payment initiation |
| **Payment flow resolution** | `PaymentFlows.hs` | Determines card vs NB vs UPI vs redirect path |

## connector-service Equivalent

In connector-service, the authorize flow is represented by:

```rust
ConnectorIntegrationV2<
    connector_flow::Authorize,        // Zero-sized flow marker
    PaymentFlowData,                  // Common flow data
    PaymentsAuthorizeData,            // Request data
    PaymentsResponseData,             // Response data
>
```

**What connector-service handles:**
- Building the gateway-specific HTTP request
- Making the API call to the gateway
- Parsing the response
- Mapping gateway status to `AttemptStatus`

**What connector-service does NOT handle (stays in euler_txn):**
- Order validation
- Gateway selection/routing
- DB state management
- Feature toggle evaluation
- ProcessTracker scheduling
- Merchant response formatting

## Migration Approach

### Phase 1 (Already Done)

The connector-call portion is already migrated via the bridge pattern:

```
euler_txn                              connector-service
---------                              -----------------
decider() -> selectGateway
  -> createTxnDetail
  -> resolvePaymentFlow
  -> shouldUseConnectorService?
     |
     YES --> buildBridgeRequest ------> ConnectorIntegrationV2<Authorize>
             parseResponse     <------  PaymentsResponseData
     |
     NO  --> Gateway/<Name>/Flow.hs (old path)
  -> updateTxnStatus
  -> scheduleSync
  -> return response
```

### Phase 2 (Future — Orchestration Migration)

When a dedicated orchestration layer is built (separate from euler_txn and connector-service):

1. **Extract `decider()` logic** into a new Authorize orchestration module
2. **Move validation logic** (order state checks, merchant eligibility)
3. **Move flow resolution** (card vs NB vs UPI routing)
4. **DB interactions** become API calls to a state service
5. **ProcessTracker scheduling** becomes async task dispatch

**Do NOT attempt Phase 2 until:**
- [ ] Phase 1 is at 100% traffic for ALL Tier 1 and Tier 2 gateways
- [ ] A state management service design is approved
- [ ] The orchestration service architecture is finalized
- [ ] Shadow traffic infrastructure can handle orchestration-level comparison

## Key Implementation Notes

### Flow Type Resolution

euler_txn resolves the payment method into a `TxnFlowType` which determines the gateway adapter path:

| `TxnFlowType` (euler_txn) | connector-service `PaymentMethodData` | Gateway Adapter Pattern |
|----------------------------|--------------------------------------|------------------------|
| `CARD_TRANSACTION` | `Card(CardData)` | Direct API call |
| `NET_BANKING` | `BankRedirect(...)` | Redirect flow |
| `INTENT` | `Upi(UpiIntent)` | Intent flow |
| `COLLECT` | `Upi(UpiCollect)` | Collect flow |
| `REDIRECT` | `BankRedirect(...)` or `Wallet(...)` | Redirect flow |
| `EMI` | `Card(CardData)` + EMI metadata | Card flow with extra params |
| `QR` | `Upi(UpiQr)` | QR code generation |

See [7.1_type_mapping.md](../7_reference/7.1_type_mapping.md) for the complete mapping.

### 3DS / Redirect Handling

For 3DS and redirect flows, euler_txn returns a redirect URL to the merchant. The bridge must correctly extract the redirect data from connector-service's response:

```rust
// connector-service response includes:
PaymentsResponseData {
    redirection_data: Some(RedirectForm {
        form_type: RedirectFormType::Uri(redirect_url),
        // ...
    }),
    // ...
}
```

```haskell
-- euler_txn bridge converts to:
PaymentResp { redirectUrl = Just (csResponse ^. _redirectUrl) }
```

### Multi-Step Authorize (Enrollment + Auth)

Some gateways (e.g., HDFC) require a two-step authorize:
1. **Enrollment** — check if card is enrolled for 3DS
2. **Authentication** — actual authorization after 3DS completion

In connector-service, this is handled by the `composite-service` crate which orchestrates multi-step flows. In euler_txn, this is inline in the gateway adapter.
