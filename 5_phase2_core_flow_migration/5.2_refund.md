# 5.2 Refund Flow Migration

> **Phase 2 — Core Transaction Flow Migration**
>
> **Prerequisites:**
> - Phase 1 Refund + RSync flows implemented for the target gateway
> - Read [4.3_per_flow_implementation.md](../4_phase1_gateway_migration/4.3_per_flow_implementation.md) for trait mapping

---

## Overview

The Refund flow handles merchant-initiated refunds — full and partial. It validates the refund request, routes to the correct gateway, creates the refund via the gateway API, and schedules async status sync.

**Priority:** P0 — refunds are critical for merchant operations and compliance.

---

## euler_txn Flow

**Entry point:** `POST /orders/:id/refunds` -> `Refund.Decider.decider`
**Source file:** `oltp/src-generated/Product/OLTP/Refund/Decider.hs`

```
HTTP Request
    |
    v
decider()
    |
    v
validateMerchantRefundEnabled     -- Check merchant config allows refunds
    |
    v
fetchOrder                         -- DB: Load OrderReference
    |
    v
validateOrderSucceeded             -- Order must be in CHARGED/SUCCESS state
    |
    v
findChargedTxn                     -- Find the TxnDetail that was charged
    |
    v
validateMandateConstraints         -- Special rules for mandate refunds
    |
    v
checkPreAuthVoid                   -- If pre-auth, void instead of refund
    |
    v
determineSplitRouting              -- Split payment routing (order-based vs txn-based)
    |
    v
createRefund (gateway adapter)     -- Call Gateway/<Name>/Flow.hs refund handler
    |
    v
scheduleRefundSync (ProcessTracker) -- Schedule async refund status check
    |
    v
return response
```

### Key Complexities

| Aspect | Detail |
|--------|--------|
| **Split payment routing** | Refunds can be order-based (split across txns) or txn-based (specific txn). Complex routing logic determines which gateway to call. |
| **Pre-auth void conversion** | If the original txn was pre-auth (AUTH only, not settled), a refund request becomes a void instead |
| **Partial refunds** | Must track cumulative refund amount vs original charge; some gateways don't support partial |
| **Mandate refund constraints** | Special validation for mandate-backed transactions |
| **Multi-gateway refunds** | An order can have charges across multiple gateways; refund must go to correct gateway |

## connector-service Equivalent

Two flows handle refunds in connector-service:

| Flow | Trait | Purpose |
|------|-------|---------|
| `Refund` | `ConnectorIntegrationV2<Refund, RefundFlowData, RefundsData, RefundsResponseData>` | Initiate refund at gateway |
| `RSync` | `ConnectorIntegrationV2<RSync, RefundFlowData, RefundSyncData, RefundsResponseData>` | Check refund status at gateway |

## Migration Approach

### Phase 1 (Gateway Call Only)

The gateway call (`createRefund`) is migrated via the bridge pattern. All validation and routing logic stays in euler_txn.

```
euler_txn                              connector-service
---------                              -----------------
decider() -> validateRefund
  -> findChargedTxn
  -> determineSplitRouting
  -> shouldUseConnectorService?
     |
     YES --> buildRefundRequest -------> ConnectorIntegrationV2<Refund>
             parseRefundResp   <-------  RefundsResponseData
     |
     NO  --> Gateway/<Name>/Flow.hs (old path)
  -> updateRefundStatus
  -> scheduleRefundSync
  -> return response
```

### Phase 2 (Future — Orchestration Migration)

When the orchestration layer moves:

1. **Extract validation logic** — merchant eligibility, order state checks, cumulative amount tracking
2. **Extract split routing** — order-based vs txn-based routing decision
3. **Pre-auth void conversion** — the logic that converts refund → void stays in the orchestrator
4. **DB interactions** — refund record creation, status updates become API calls

**Do NOT attempt Phase 2 for refunds until:**
- [ ] Phase 1 refund is at 100% for all Tier 1/2 gateways
- [ ] Authorize flow Phase 2 migration is complete (refund depends on authorize state)
- [ ] Split payment routing edge cases are fully documented

## Key Implementation Notes

### Refund Status Mapping

| euler_txn Refund Status | connector-service `RefundStatus` |
|------------------------|--------------------------------|
| `SUCCESS` / `REFUND_MANUAL_REVIEW` | `Success` |
| `PENDING` | `Pending` |
| `FAILURE` | `Failure` |
| `INITIATED` | `Pending` |

### Partial Refund Handling

euler_txn tracks refund amounts in the `Refund` DB table:

```haskell
-- euler_txn creates a Refund record with:
Refund { amount = refundAmount, orderId = orderId, txnDetailId = txnId, ... }
```

connector-service receives the amount in `RefundsData`:

```rust
pub struct RefundsData {
    pub refund_id: String,
    pub payment_amount: i64,      // Original charge amount
    pub refund_amount: i64,       // Amount to refund
    pub currency: Currency,
    // ...
}
```

The bridge must correctly map `refundAmount` from euler_txn to `refund_amount` in connector-service.

### Gateway-Specific Refund Behavior

Some gateways require the original transaction ID to process refunds:

| Gateway | Requires Original Txn ID? | Field in euler_txn | Field in connector-service |
|---------|--------------------------|-------------------|--------------------------|
| HDFC | Yes | `epgTxnId` from TxnDetail | `connector_transaction_id` in RefundsData |
| Paytm | Yes | `successResponseId` | `connector_transaction_id` |
| Razorpay | Yes | `epgTxnId` | `connector_transaction_id` |
| Billdesk | Yes | Gateway-specific ID | `connector_transaction_id` |
