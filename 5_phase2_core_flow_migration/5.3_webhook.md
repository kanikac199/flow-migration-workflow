# 5.3 Webhook Flow Migration

> **Phase 2 — Core Transaction Flow Migration**
>
> **Prerequisites:**
> - Phase 1 gateway connector is live for the target gateway
> - Understand the gateway's webhook format and authentication mechanism

---

## Overview

The Webhook flow handles inbound notifications from payment gateways — payment status updates, refund confirmations, chargeback notifications, etc. euler_txn receives the raw webhook, authenticates it, parses the gateway-specific payload, maps it to internal events, and updates transaction state.

**Priority:** P2 — webhooks are important but can lag behind authorize/refund migration since sync flows cover status updates.

---

## euler_txn Flow

**Entry point:** `POST /v2/pay/webhooks/*` -> `WebhookFlow`
**Source file:** `Product/OLTP/WebhookFlow`

```
Gateway POST (webhook notification)
    |
    v
authenticate webhook               -- Verify signature/IP/secret
    |
    v
parse gateway-specific payload      -- Each gateway has its own webhook format
    |
    v
map to internal event               -- Determine event type: payment success, refund, etc.
    |
    v
update TxnDetail status             -- DB: Update transaction state
    |
    v
trigger merchant webhook            -- Outbound: notify merchant's webhook URL
```

### Key Aspects

| Aspect | Detail |
|--------|--------|
| **Authentication** | Each gateway uses a different mechanism: HMAC signature, IP whitelisting, shared secret in payload, certificate verification |
| **Idempotency** | Webhooks can arrive multiple times; must be idempotent |
| **Ordering** | Webhooks may arrive out of order; must handle gracefully |
| **Outbound notification** | After processing the inbound gateway webhook, euler_txn fires an outbound webhook to the merchant |

## connector-service Equivalent

connector-service has built-in webhook support via two traits:

### IncomingWebhook Trait

```rust
// Handles parsing and verification of gateway webhooks
pub trait IncomingWebhook {
    fn get_webhook_source_verification_algorithm(
        &self,
        request: &IncomingWebhookRequestDetails,
    ) -> CustomResult<Box<dyn crypto::VerifySignature + Send>, ConnectorError>;

    fn get_webhook_source_verification_signature(
        &self,
        request: &IncomingWebhookRequestDetails,
        connector_webhook_secrets: &api_models::webhooks::ConnectorWebhookSecrets,
    ) -> CustomResult<Vec<u8>, ConnectorError>;

    fn get_webhook_source_verification_message(
        &self,
        request: &IncomingWebhookRequestDetails,
        merchant_id: &str,
        secret: &[u8],
    ) -> CustomResult<Vec<u8>, ConnectorError>;

    fn get_webhook_object_reference_id(
        &self,
        request: &IncomingWebhookRequestDetails,
    ) -> CustomResult<ObjectReferenceId, ConnectorError>;

    fn get_webhook_event_type(
        &self,
        request: &IncomingWebhookRequestDetails,
    ) -> CustomResult<IncomingWebhookEvent, ConnectorError>;

    fn get_webhook_resource_object(
        &self,
        request: &IncomingWebhookRequestDetails,
    ) -> CustomResult<Box<dyn masking::ErasedMaskSerialize>, ConnectorError>;
}
```

## Migration Approach

### What Moves to connector-service

| Component | Migrates? | Notes |
|-----------|-----------|-------|
| Webhook signature verification | Yes | Implement `IncomingWebhook` trait per connector |
| Gateway payload parsing | Yes | Parse gateway-specific JSON/XML into structured types |
| Event type mapping | Yes | Map gateway events to `IncomingWebhookEvent` enum |
| Transaction reference extraction | Yes | Extract order_id/txn_id from webhook payload |

### What Stays in euler_txn

| Component | Why |
|-----------|-----|
| DB state updates | connector-service is stateless |
| Outbound merchant webhook | Business logic + merchant config dependent |
| Idempotency checks | Requires DB lookup of previous webhook processing |
| Event deduplication | Requires state (Redis/DB) |

### Bridge Pattern for Webhooks

```
Gateway                     euler_txn                    connector-service
-------                     ---------                    -----------------
  |                             |                              |
  |--- Webhook POST ---------->|                              |
  |                             |                              |
  |                             |--- Forward raw body -------->|
  |                             |                              |
  |                             |                    Verify signature
  |                             |                    Parse payload
  |                             |                    Map event type
  |                             |                    Extract references
  |                             |                              |
  |                             |<-- Parsed webhook response --|
  |                             |                              |
  |                       Update DB state                      |
  |                       Fire merchant webhook                |
  |                             |                              |
```

## Key Implementation Notes

### Webhook Event Type Mapping

| euler_txn Event | connector-service `IncomingWebhookEvent` |
|----------------|----------------------------------------|
| Payment success | `PaymentIntentSuccess` |
| Payment failure | `PaymentIntentFailure` |
| Refund success | `RefundSuccess` |
| Refund failure | `RefundFailure` |
| Chargeback | `DisputeOpened` |
| Mandate confirmed | `MandateConfirmed` |

### Authentication Methods per Gateway

| Gateway | Verification Method | euler_txn Implementation | connector-service Implementation |
|---------|-------------------|------------------------|--------------------------------|
| Razorpay | HMAC-SHA256 signature | Header verification in webhook handler | `get_webhook_source_verification_algorithm()` returns HMAC |
| Paytm | Checksum in payload | Custom checksum validation | Same via trait |
| HDFC | IP whitelisting + response params | IP check + param verification | Signature-based verification |
| Stripe | HMAC-SHA256 (`Stripe-Signature` header) | Already in connector-service | `IncomingWebhook` trait implemented |
