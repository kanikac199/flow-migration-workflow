# 5.4 Mandate Flow Migration

> **Phase 2 — Core Transaction Flow Migration**
>
> **Prerequisites:**
> - Phase 1 gateway connector is live for the target gateway
> - Understand the gateway's mandate (recurring payment) support

---

## Overview

The Mandate flow handles recurring payment agreements — registering a mandate with a gateway, executing recurring charges against it, and revoking mandates. This includes eNACH, eMandate, UPI autopay, and card-on-file mandates.

**Priority:** P1 — mandates are critical for subscription businesses.

---

## euler_txn Flow

**Source file:** `Product/OLTP/Mandate.hs`

### Mandate Register

```
HTTP Request (mandate registration)
    |
    v
createMandate                       -- DB: Create Mandate record
    |
    v
initiateTxn (with mandate params)   -- Gateway call with mandate setup data
    |
    v
handleResponse                      -- Parse gateway response
    |
    v
updateMandateStatus                 -- DB: Update Mandate with gateway mandate ID
```

### Mandate Execute (Recurring Charge)

```
HTTP Request (POST /mandates/:id)
    |
    v
loadMandate                          -- DB: Load Mandate record
    |
    v
processMandateExecution              -- Gateway call for recurring charge
    |
    v
handleResponse                       -- Parse gateway response
    |
    v
updateTxnStatus                      -- DB: Update TxnDetail
```

### Mandate Revoke

```
HTTP Request (revoke mandate)
    |
    v
loadMandate                          -- DB: Load Mandate record
    |
    v
revokeMandate                        -- Gateway call to cancel mandate
    |
    v
updateMandateStatus                  -- DB: Mark mandate as revoked
```

| Mandate Register | `SetupMandate` | `ConnectorIntegrationV2<SetupMandate, PaymentFlowData, SetupMandateRequestData, PaymentsResponseData>` |

## Migration Approach

### What Moves to connector-service

| Component | Migrates? | Notes |
|-----------|-----------|-------|
| Mandate registration gateway call | Yes | `SetupMandate` flow builds registration request |
| Recurring charge gateway call | Yes | `RepeatPayment` flow executes charge using stored mandate |
| Mandate revocation gateway call | Yes | `MandateRevoke` flow cancels the mandate |
| Gateway mandate ID extraction | Yes | Parse mandate token/ID from gateway response |

### What Stays in euler_txn

| Component | Why |
|-----------|-----|
| Mandate DB record management | connector-service is stateless |
| Mandate validation (amount limits, frequency) | Business logic |
| eNACH/eMandate flow orchestration | Multi-step with bank redirects and async confirmation |
| UPI autopay notification handling | Involves merchant notification + DB state |
| Mandate-refund constraint validation | Cross-entity business rule |

### Bridge Pattern for Mandates

```
euler_txn                              connector-service
---------                              -----------------
createMandate (DB)
  -> buildMandateRequest -----------> ConnectorIntegrationV2<SetupMandate>
     parseMandateResp   <-----------  PaymentsResponseData (with mandate_id)
  -> updateMandate (DB) with gateway mandate ID

-- Later, for recurring charge: --

loadMandate (DB)
  -> buildRepeatRequest -----------> ConnectorIntegrationV2<RepeatPayment>
     parseRepeatResp    <-----------  PaymentsResponseData
  -> updateTxnDetail (DB)
```

## Key Implementation Notes

### Mandate Types

| euler_txn | `TxnObjectType` | connector-service Flow |
|-----------|-----------------|----------------------|
| Card mandate register | `MANDATE_REGISTER` | `SetupMandate` |
| Card mandate payment | `MANDATE_PAYMENT` | `RepeatPayment` |
| eNACH register | `EMANDATE_REGISTER` | `SetupMandate` (eMandate variant) |
| eNACH payment | `EMANDATE_PAYMENT` | `RepeatPayment` (eMandate variant) |

### Mandate Data in connector-service

```rust
pub struct SetupMandateRequestData {
    pub mandate_id: Option<String>,
    pub amount: i64,
    pub currency: Currency,
    pub payment_method_data: PaymentMethodData,
    pub setup_future_usage: Option<FutureUsage>,
    pub mandate_type: Option<MandateType>,
    // ...
}

pub struct RepeatPaymentData {
    pub mandate_id: String,            // Gateway mandate ID from registration
    pub amount: i64,
    pub currency: Currency,
    pub connector_mandate_id: Option<String>,
    // ...
}
```

### Gateway-Specific Mandate Behavior

| Gateway | Mandate Type | Special Handling |
|---------|-------------|-----------------|
| HDFC | Card-on-file (SI) | Two-phase: registration + first charge |
| Paytm | UPI autopay, SI | UPI mandate has async confirmation flow |
| Razorpay | UPI autopay | Mandate confirmation via webhook |
| ICICI | eNACH | Bank redirect for mandate signing |
| SBI | eNACH | Similar to ICICI with different API format |
