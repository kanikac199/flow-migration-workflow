# 8.3 Rollback Plan

> **Testing & Operations — Emergency and Planned Rollback**
>
> Every migrated flow is behind a feature flag. Rollback = flip the flag.

---

## Rollback Trigger Conditions

Immediately rollback if **any** of these conditions are met:

| Condition | Threshold |
|-----------|-----------|
| Error rate increase | > 0.5% compared to baseline |
| p99 latency increase | > 200ms compared to baseline |
| Success rate drop | > 0.1% compared to baseline |
| Data inconsistency | Any status mismatch or amount mismatch detected |
| Gateway rejection spike | Unusual increase in gateway-side rejections |
| Connector-service unavailability | > 30 seconds of 5xx errors or timeouts |

---

## 11.1 Instant Rollback (Feature Flag)

### Procedure

1. **Flip feature flag:** `CONNECTOR_SERVICE_<GATEWAY>_ENABLED = false`
   - This can be a ServiceConfiguration DB update, Redis key, or env var — depends on implementation
   - Change takes effect on next request (no restart needed if config is hot-reloaded)

2. **Verify traffic is flowing through old path:**
   - Check metrics dashboard: connector-service request count should drop to 0 for this gateway
   - euler_txn's direct gateway adapter requests should increase back to baseline

3. **Investigate root cause:**
   - Check connector-service logs for errors
   - Check euler_txn bridge logs for request/response discrepancies
   - Check gateway dashboard for any issues on their side

4. **Fix in connector-service:**
   - Identify the bug (status mapping, request format, auth, etc.)
   - Fix and deploy connector-service
   - Verify with unit tests

5. **Re-enable gradually:**
   - Start from shadow mode again
   - Confirm the fix resolves the issue
   - Resume gradual rollout from the last stable percentage

### Feature Flag Hierarchy

```
Global kill switch (highest priority):
  CONNECTOR_SERVICE_GLOBAL_ENABLED = false  → ALL gateways roll back

Per-gateway toggle:
  CONNECTOR_SERVICE_HDFC_ENABLED = false    → Only HDFC rolls back

Per-merchant toggle (lowest priority):
  CONNECTOR_SERVICE_<GATEWAY>_<MERCHANT_ID> = false  → Only this merchant rolls back
```

### Rollback Speed

| Method | Time to Effect | Restart Required? |
|--------|---------------|-------------------|
| Redis key update | < 1 second | No |
| DB ServiceConfiguration update | < 5 seconds (with cache TTL) | No |
| Env var change + restart | 1-5 minutes | Yes |
| Config file change + deploy | 5-15 minutes | Yes |

**Recommendation:** Use Redis-backed feature flags for instant rollback capability.

## 11.2 Data Consistency on Rollback

### Phase 1 (Stateless — No Data Impact)

Since connector-service is **stateless** in Phase 1, rollback has **no data implications**:
- euler_txn owns all state (DB records)
- connector-service only processes the gateway API call
- Flipping the flag simply routes the gateway call back to euler_txn's built-in adapter
- No orphaned records, no state inconsistency

### Phase 2+ (If Orchestration Moves)

If Phase 2 migration is underway and orchestration logic has moved, ensure on rollback:

- [ ] Transaction state in DB is consistent regardless of which path processed it
- [ ] No orphaned transactions in intermediate states
- [ ] ProcessTracker tasks can be picked up by either path
- [ ] In-flight transactions at the moment of rollback are handled:
  - Transactions initiated via connector-service but not yet completed should be sync-able via both paths
  - The `connector_transaction_id` stored by connector-service must be accessible to euler_txn's old path for status sync

### Phase 3 (Async Workflows)

For async workflow rollback (TxnSync, Refund sync, etc.):

- ProcessTracker tasks in `PENDING` state will be picked up by the old workflow path automatically after rollback
- Tasks currently `RUNNING` via the new path should complete (or timeout and retry via old path)
- No special data migration needed — ProcessTracker state is agnostic to which path executes the workflow

## Rollback Runbook

### Step-by-step

```
1. DETECT
   └── Alert fires (error rate, latency, success rate)
   └── On-call engineer confirms the issue is related to connector-service migration

2. ROLLBACK
   └── Set feature flag: CONNECTOR_SERVICE_<GATEWAY>_ENABLED = false
   └── Time: < 1 minute

3. VERIFY
   └── Check metrics: connector-service traffic for this gateway = 0
   └── Check metrics: euler_txn direct gateway traffic = back to baseline
   └── Check: no increase in errors on the old path
   └── Time: < 5 minutes

4. COMMUNICATE
   └── Notify team in incident channel
   └── Document: which gateway, what time, what triggered the rollback

5. INVESTIGATE
   └── Pull connector-service logs for the affected time window
   └── Pull euler_txn bridge logs
   └── Identify root cause
   └── Time: hours to days

6. FIX & RE-DEPLOY
   └── Fix the issue in connector-service
   └── Add regression test
   └── Deploy connector-service
   └── Resume from shadow mode
```

---

## Post-Rollback Checklist

- [ ] Feature flag confirmed OFF
- [ ] Metrics confirm traffic is on old path
- [ ] No elevated error rate on old path
- [ ] Incident documented
- [ ] Root cause identified
- [ ] Fix implemented and tested
- [ ] Regression test added
- [ ] Re-rollout plan created
