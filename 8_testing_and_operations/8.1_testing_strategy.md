# 8.1 Testing Strategy

> **Testing & Operations**
>
> **Prerequisites:**
> - Connector implementation complete in connector-service
> - Bridge wired in euler_txn

---

## Overview

Every migrated gateway/flow must pass through a rigorous testing pipeline before production traffic is routed to the new path. The testing strategy has three layers: unit tests, integration/shadow testing, and canary deployment.

---

## 8.1.1 Unit Testing (connector-service)

For every connector implementation, add tests in `backend/ucs-connector-tests/`.

### Request Body Tests

```rust
#[test]
fn test_<connector>_authorize_request_body() {
    // Given: RouterDataV2 with known values (mirror a real euler_txn request)
    // When: TryFrom converts to connector request
    // Then: Assert exact field values match what euler_txn would send
}
```

**What to verify:**
- All required fields are present
- Amount format matches gateway's expectation (minor units vs major units)
- Currency is correctly formatted
- Auth credentials are correctly placed (header vs body)
- Optional fields are handled (present when available, absent when not)

### Response Mapping Tests

```rust
#[test]
fn test_<connector>_authorize_response_mapping() {
    // Given: Raw gateway response JSON (captured from production)
    // When: handle_response_v2 processes it
    // Then: Assert status mapping matches euler_txn's behavior
}
```

**What to verify:**
- Every gateway status code maps to the correct `AttemptStatus`
- `connector_transaction_id` is correctly extracted
- Redirect URLs are correctly parsed (for 3DS/redirect flows)
- Amount in response matches expectation

### Error Response Tests

```rust
#[test]
fn test_<connector>_error_response_mapping() {
    // Given: Gateway error response
    // When: get_error_response processes it
    // Then: Assert error code/message extraction matches euler_txn
}
```

**What to verify:**
- Error code maps to `bankErrorCode` equivalent
- Error message maps to `bankErrorMessage` equivalent
- HTTP error statuses (4xx, 5xx) are handled correctly
- Malformed responses don't panic (return proper error)

### Test Data Source

Capture production request/response pairs from euler_txn logs. For each gateway being migrated, collect:

1. Successful authorize request + response
2. Failed authorize (declined) request + response
3. 3DS/redirect authorize request + response
4. Capture request + response
5. Refund request + response
6. Sync (PSync) request + response
7. Error scenarios (timeout, gateway down, malformed response)

## 8.1.2 Integration Testing (Shadow Traffic)

### Shadow Traffic Architecture

```
                        ┌──────────────────────────┐
                        │     euler_txn (old path)  │──► Gateway ──► Response A
                        │                           │
Real Request ──────────►│                           │
                        │     connector-service     │──► Gateway ──► Response B
                        │     (new path, shadow)    │
                        └──────────────────────────┘
                                     │
                              Compare A vs B
                              Log discrepancies
```

### Implementation

1. In euler_txn, call **BOTH** paths (old gateway adapter + new connector-service bridge)
2. Use the **OLD** path's response as the actual response (merchant never sees the new path)
3. Log connector-service's response for comparison
4. Compare: status code, response body structure, status mapping, error handling
5. Alert on any discrepancy

### Critical Comparison Points

| Field | Source (euler_txn) | Target (connector-service) | Must Match? |
|-------|-------------------|---------------------------|-------------|
| Transaction status | `TxnStatus` from response | `AttemptStatus` from response | **Exact** (via mapping) |
| Gateway txn ID | `epgTxnId` / `successResponseId` | `resource_id.connector_transaction_id` | **Exact** |
| Error code | `bankErrorCode` | `ErrorResponse.code` | **Exact** |
| Error message | `bankErrorMessage` | `ErrorResponse.message` | Best effort |
| Redirect URL | `redirectUrl` in PaymentResp | `redirection_data.form_type.uri` | **Exact** |
| Amount | From gateway response | From gateway response | **Exact** |

### Shadow Mode Duration

Run shadow mode for **at least 1 week** before moving to canary. During this time:
- Fix all discrepancies found
- Verify all edge cases are covered
- Confirm latency is within acceptable bounds

## 8.1.3 Canary Deployment

### Rollout Schedule

```
Week 1: Shadow mode (0% real traffic through new path)
  └── Fix all discrepancies found

Week 2: 1% traffic via feature flag
  └── Monitor error rates, latency, success rates

Week 3: 10% traffic
  └── Validate at scale, check edge cases

Week 4: 50% traffic
  └── Full production validation

Week 5: 100% traffic
  └── Old path remains as fallback

Week 7+: Remove old path code
```

### Monitoring Metrics

At each rollout stage, monitor:

| Metric | Old Path Baseline | Acceptable Variance |
|--------|-------------------|-------------------|
| Success rate | Established | < 0.1% drop |
| Error rate | Established | < 0.5% increase |
| p50 latency | Established | < 10ms increase |
| p95 latency | Established | < 30ms increase |
| p99 latency | Established | < 50ms increase |
| Timeout rate | Established | No increase |

### Rollback Triggers

Immediately rollback (flip feature flag) if:
- Error rate increases > 0.5% compared to baseline
- p99 latency increases > 200ms compared to baseline
- Success rate drops > 0.1% compared to baseline
- Any data inconsistency detected (status mismatch, amount mismatch)

See [8.3_rollback_plan.md](8.3_rollback_plan.md) for detailed rollback procedure.

## 8.1.4 Testing Async Workflows

For TxnSync and other async workflows (Phase 3), additional testing is needed:

1. **Retry behavior test:** Simulate a pending payment, verify TxnSync picks it up and resolves it
2. **Connector-service failure test:** Simulate connector-service being unavailable, verify workflow retries correctly
3. **Status transition test:** Verify the workflow produces the same final transaction status via both old and new paths
4. **Lock contention test:** Verify the distributed lock mechanism works correctly with the new path
5. **Timing test:** Verify retry schedules are not disrupted by additional bridge latency

## 8.1.5 gRPC Smoke Test (Local Runtime Validation)

Before moving to integration/shadow testing, validate the full gRPC pipeline locally by sending a real request through the connector-service to the connector's preprod/sandbox environment.

### Prerequisites

- `grpcurl` installed (`brew install grpcurl`)
- Connector's preprod/UAT credentials (merchant_id, API keys, etc.)
- The connector's preprod/sandbox base URL configured in `config/development.toml`

### Step 1: Build and Start the Server

```bash
export PATH="$HOME/.cargo/bin:$PATH"
cargo build -p grpc-server
RUN_ENV=development nohup ./target/debug/grpc-server > /tmp/grpc-server.log 2>&1 &
echo "Server PID: $!"
sleep 5
# Verify it's running (listens on port 8000 for gRPC, 8080 for metrics)
curl -s http://localhost:8080/health || echo "Server not ready"
```

### Step 2: Construct the `x-connector-config` Header

The header is parsed via `serde_json::from_str` (NOT protobuf JSON). Format:

```json
{"config":{"<VariantName>":{"field1":"value1","field2":"value2"}}}
```

**Key rules:**
- The variant name matches the Rust enum variant exactly (PascalCase, e.g., `Phonepe`, `Stripe`, `Adyen`)
- `Secret<String>` fields are **plain strings** (NOT `{"value":"..."}`)
- The outer key is always `"config"`

**Find the correct variant name:** Check the generated proto code or `backend/domain_types/src/router_data.rs` for the `ConnectorSpecificConfig` enum variant.

**Example (PhonePe):**
```
x-connector-config: {"config":{"Phonepe":{"merchant_id":"JUSPAYUAT","salt_key":"<salt>","salt_index":"1"}}}
```

### Step 3: Construct the Request Body

The body uses **protobuf JSON encoding** (NOT serde). Key difference:

- `SecretString` proto message has `message SecretString { string value = 1; }`, so in protobuf JSON it's `{"value":"..."}`
- Enums are string names
- Standard protobuf JSON rules apply

**Example (TriggerOtpForWallet):**
```json
{
  "mobile_number": {"value": "8178392166"},
  "merchant_id": {"value": "JUSPAYUAT"}
}
```

### Step 4: Send the gRPC Request

```bash
grpcurl -plaintext \
  -H 'x-connector-config: <JSON from Step 2>' \
  -H 'x-merchant-id: <merchant_id>' \
  -H 'x-tenant-id: default' \
  -H 'x-request-id: test_001' \
  -H 'x-connector-request-reference-id: test_ref_001' \
  -d '<JSON from Step 3>' \
  localhost:8000 types.PaymentService/<RpcMethod>
```

**Required headers:**
| Header | Value |
|--------|-------|
| `x-connector-config` | Connector-specific config JSON (Step 2) |
| `x-merchant-id` | Merchant identifier |
| `x-tenant-id` | Tenant identifier (use `default` for testing) |
| `x-request-id` | Unique request ID for tracing |
| `x-connector-request-reference-id` | Reference ID for connector correlation |

**Find the RPC method name:** Check `backend/grpc-api-types/proto/services.proto` for the service and method name.

### Step 5: Validate the Response

The response should contain:

1. **`rawConnectorRequest`**: Verify the outgoing URL, headers, and body match the Haskell implementation
2. **`rawConnectorResponse`**: The raw response from the connector
3. **Business fields**: Extracted response data (e.g., `otpToken`, `connectorTransactionId`)
4. **`statusCode`**: HTTP status from the connector

**Validation checklist:**
- [ ] URL in `rawConnectorRequest` points to the correct connector endpoint
- [ ] Request headers match the Haskell implementation (e.g., `X-VERIFY`, `Authorization`)
- [ ] Request body matches the Haskell implementation (decode Base64 payloads if applicable)
- [ ] Connector returned a well-formed response (success or a recognized business error)
- [ ] Response fields are correctly extracted into the gRPC response message
- [ ] If connector returns success, the business data is correct

**Success criteria:**
- A **200 with correct business data** = full success
- A **4xx with a well-formed connector error** (e.g., `INVALID_MERCHANT_ID`, `Api Mapping Not Found`) = pipeline works, check credentials/config
- A **connection error or parse failure** = code bug, investigate

### Step 6: Verify Checksums/Signatures Independently

For connectors using request signing (e.g., PhonePe X-VERIFY, Razorpay signature):

```bash
# Example: Verify PhonePe X-VERIFY checksum
BASE64='<base64 payload from rawConnectorRequest>'
ENDPOINT='/v3/merchant/otp/send'
SALT='<salt_key>'
HASH=$(printf "%s%s%s" "$BASE64" "$ENDPOINT" "$SALT" | shasum -a 256 | cut -d' ' -f1)
echo "Computed: ${HASH}###<salt_index>"
# Compare with X-VERIFY header in rawConnectorRequest
```

### Common Issues

| Symptom | Cause | Fix |
|---------|-------|-----|
| `Failed to parse X-Connector-Config JSON` | Wrong JSON format for header | Use serde format: `{"config":{"VariantName":{...}}}` with plain strings for `Secret<String>` |
| `X-Connector-Config header missing config` | Missing `config` key or wrong variant name | Check exact enum variant name in generated code |
| `InvalidConnectorConfig: secondary_base_url` | Flow needs `secondary_base_url` but not configured | Add `<connector>.secondary_base_url` to `config/development.toml` |
| `Api Mapping Not Found` (PhonePe) | Wrong base URL domain for the API category | PhonePe has separate domains for PG (Hermes) vs Wallet (Mercury/pg-sandbox). Use `secondary_base_url` for wallet flows |
| Connection refused on localhost:8000 | Server not started or crashed | Check `/tmp/grpc-server.log` for errors; ensure ports 8000 and 8080 are free |

### PhonePe-Specific: Dual Base URL Architecture

PhonePe uses **two separate API gateways**:

| Gateway | Preprod Domain | Production Domain | APIs |
|---------|---------------|-------------------|------|
| **Hermes** (PG) | `api-preprod.phonepe.com/apis/hermes/` | `api.phonepe.com/apis/hermes/` | Pay, status, refund, VPA validate |
| **Mercury/pg-sandbox** (Wallet) | `api-preprod.phonepe.com/apis/pg-sandbox/` | `mercury-t2.phonepe.com/` | OTP send/verify, wallet balance/debit/topup, delink |

Configure in TOML:
```toml
phonepe.base_url = "https://api-preprod.phonepe.com/apis/hermes/"           # PG flows
phonepe.secondary_base_url = "https://api-preprod.phonepe.com/apis/pg-sandbox/" # Wallet flows
```

Wallet flow implementations must use `secondary_base_url` in their `get_url()`:
```rust
fn get_url(&self, req: &RouterDataV2<...>) -> CustomResult<String, errors::ConnectorError> {
    let secondary_base_url = req
        .resource_common_data
        .connectors
        .phonepe
        .secondary_base_url
        .as_ref()
        .ok_or(errors::ConnectorError::InvalidConnectorConfig {
            config: "secondary_base_url",
        })?;
    Ok(format!("{}{}", secondary_base_url, constants::API_OTP_SEND_ENDPOINT))
}
```
