# 8.1 Testing Strategy

> **Testing & Operations**
>
> **Prerequisites:**
> - Connector implementation complete in connector-service
> - Bridge wired in euler_txn

---

## Overview

Every migrated gateway/flow must pass through a rigorous testing pipeline before production traffic is routed to the new path. The testing strategy has three layers: unit tests, integration/shadow testing, and canary deployment.

---

## 8.1.1 Unit Testing (connector-service)

For every connector implementation, add tests in `backend/ucs-connector-tests/`.

### Request Body Tests

```rust
#[test]
fn test_<connector>_authorize_request_body() {
    // Given: RouterDataV2 with known values (mirror a real euler_txn request)
    // When: TryFrom converts to connector request
    // Then: Assert exact field values match what euler_txn would send
}
```

**What to verify:**
- All required fields are present
- Amount format matches gateway's expectation (minor units vs major units)
- Currency is correctly formatted
- Auth credentials are correctly placed (header vs body)
- Optional fields are handled (present when available, absent when not)

### Response Mapping Tests

```rust
#[test]
fn test_<connector>_authorize_response_mapping() {
    // Given: Raw gateway response JSON (captured from production)
    // When: handle_response_v2 processes it
    // Then: Assert status mapping matches euler_txn's behavior
}
```

**What to verify:**
- Every gateway status code maps to the correct `AttemptStatus`
- `connector_transaction_id` is correctly extracted
- Redirect URLs are correctly parsed (for 3DS/redirect flows)
- Amount in response matches expectation

### Error Response Tests

```rust
#[test]
fn test_<connector>_error_response_mapping() {
    // Given: Gateway error response
    // When: get_error_response processes it
    // Then: Assert error code/message extraction matches euler_txn
}
```

**What to verify:**
- Error code maps to `bankErrorCode` equivalent
- Error message maps to `bankErrorMessage` equivalent
- HTTP error statuses (4xx, 5xx) are handled correctly
- Malformed responses don't panic (return proper error)

### Test Data Source

Capture production request/response pairs from euler_txn logs. For each gateway being migrated, collect:

1. Successful authorize request + response
2. Failed authorize (declined) request + response
3. 3DS/redirect authorize request + response
4. Capture request + response
5. Refund request + response
6. Sync (PSync) request + response
7. Error scenarios (timeout, gateway down, malformed response)

## 8.1.2 Integration Testing (Shadow Traffic)

### Shadow Traffic Architecture

```
                        ┌──────────────────────────┐
                        │     euler_txn (old path)  │──► Gateway ──► Response A
                        │                           │
Real Request ──────────►│                           │
                        │     connector-service     │──► Gateway ──► Response B
                        │     (new path, shadow)    │
                        └──────────────────────────┘
                                     │
                              Compare A vs B
                              Log discrepancies
```

### Implementation

1. In euler_txn, call **BOTH** paths (old gateway adapter + new connector-service bridge)
2. Use the **OLD** path's response as the actual response (merchant never sees the new path)
3. Log connector-service's response for comparison
4. Compare: status code, response body structure, status mapping, error handling
5. Alert on any discrepancy

### Critical Comparison Points

| Field | Source (euler_txn) | Target (connector-service) | Must Match? |
|-------|-------------------|---------------------------|-------------|
| Transaction status | `TxnStatus` from response | `AttemptStatus` from response | **Exact** (via mapping) |
| Gateway txn ID | `epgTxnId` / `successResponseId` | `resource_id.connector_transaction_id` | **Exact** |
| Error code | `bankErrorCode` | `ErrorResponse.code` | **Exact** |
| Error message | `bankErrorMessage` | `ErrorResponse.message` | Best effort |
| Redirect URL | `redirectUrl` in PaymentResp | `redirection_data.form_type.uri` | **Exact** |
| Amount | From gateway response | From gateway response | **Exact** |

### Shadow Mode Duration

Run shadow mode for **at least 1 week** before moving to canary. During this time:
- Fix all discrepancies found
- Verify all edge cases are covered
- Confirm latency is within acceptable bounds

## 8.1.3 Canary Deployment

### Rollout Schedule

```
Week 1: Shadow mode (0% real traffic through new path)
  └── Fix all discrepancies found

Week 2: 1% traffic via feature flag
  └── Monitor error rates, latency, success rates

Week 3: 10% traffic
  └── Validate at scale, check edge cases

Week 4: 50% traffic
  └── Full production validation

Week 5: 100% traffic
  └── Old path remains as fallback

Week 7+: Remove old path code
```

### Monitoring Metrics

At each rollout stage, monitor:

| Metric | Old Path Baseline | Acceptable Variance |
|--------|-------------------|-------------------|
| Success rate | Established | < 0.1% drop |
| Error rate | Established | < 0.5% increase |
| p50 latency | Established | < 10ms increase |
| p95 latency | Established | < 30ms increase |
| p99 latency | Established | < 50ms increase |
| Timeout rate | Established | No increase |

### Rollback Triggers

Immediately rollback (flip feature flag) if:
- Error rate increases > 0.5% compared to baseline
- p99 latency increases > 200ms compared to baseline
- Success rate drops > 0.1% compared to baseline
- Any data inconsistency detected (status mismatch, amount mismatch)

See [8.3_rollback_plan.md](8.3_rollback_plan.md) for detailed rollback procedure.

## 8.1.4 Testing Async Workflows

For TxnSync and other async workflows (Phase 3), additional testing is needed:

1. **Retry behavior test:** Simulate a pending payment, verify TxnSync picks it up and resolves it
2. **Connector-service failure test:** Simulate connector-service being unavailable, verify workflow retries correctly
3. **Status transition test:** Verify the workflow produces the same final transaction status via both old and new paths
4. **Lock contention test:** Verify the distributed lock mechanism works correctly with the new path
5. **Timing test:** Verify retry schedules are not disrupted by additional bridge latency

## 8.1.5 gRPC Smoke Test (Local Runtime Validation)

Before moving to integration/shadow testing, validate the full gRPC pipeline locally by sending a real request through the connector-service to the connector's preprod/sandbox environment.

### Prerequisites

- `grpcurl` installed (`brew install grpcurl`) — verify with `which grpcurl`
- Connector's preprod/UAT credentials (merchant_id, API keys, etc.)
- The connector's preprod/sandbox base URL configured in `config/development.toml`
- All pre-call dependency flows implemented (see Step 0)

### Step 0: Identify and Execute Pre-call Dependencies

Before testing the target flow, determine if it requires inputs from prior flows.

**Pre-call analysis:**
1. Does this flow require IDs, tokens, or references created by another flow?
   - Status-check flows need the entity to exist (e.g., MandateStatusCheck needs
     a mandate created via SetupMandate)
   - Wallet flows may need auth tokens from OTP verification
   - Sync flows need a transaction/refund to have been initiated
   - Debit flows need a funded wallet or authorized mandate
2. Are those pre-call flows implemented in connector-service for this connector?
   - If YES (real implementation): execute them via grpcurl to produce the required inputs
   - If YES (empty stub) or NO: go back to implementation — these must be built first

**Pre-call execution order:**
- Execute pre-calls via grpcurl in dependency order
- Capture output IDs/tokens from each response
- Use those outputs as inputs for subsequent calls
- Document the full chain in the test report

**ZERO TOLERANCE: Do NOT skip pre-calls and use dummy values.** If a flow needs
a token, ID, or reference from a prior flow, that prior flow MUST be implemented
as a real connector integration (not an empty stub) and executed via grpcurl to
produce the real value. This is an absolute rule with NO exceptions:
- If the pre-call flow exists but is an empty stub in connector-service for this
  connector: go back to implementation and build the real integration first.
- If the pre-call flow type doesn't exist at all: go back and create it (all 7 layers).
- NEVER use a hardcoded, dummy, or placeholder value for any field that should
  come from a pre-call output. This produces false test results and is the #1
  cause of wasted debugging time.
- A test that uses dummy values is INVALID regardless of the HTTP status code
  returned. Even if you get a 200, the test is not trustworthy.

### Step 1: Lint, Format, and Build the Server

**Before building, always run clippy and fmt first:**

```bash
export PATH="$HOME/.cargo/bin:$PATH"

# Lint — must pass with zero errors/warnings
cargo clippy --all-targets --all-features

# Format — must produce no changes
cargo +nightly fmt --all

# Build
cargo build -p grpc-server
RUN_ENV=development nohup ./target/debug/grpc-server > /tmp/grpc-server.log 2>&1 &
echo "Server PID: $!"
sleep 5
# Verify it's running (listens on port 8000 for gRPC, 8080 for metrics)
# The health endpoint returns an empty 200 response on success
ps -p $! -o pid,command || echo "Server not ready — check /tmp/grpc-server.log"
```

### Step 2: Construct the `x-connector-config` Header

The header is parsed via `serde_json::from_str` (NOT protobuf JSON). Format:

```json
{"config":{"<VariantName>":{"field1":"value1","field2":"value2"}}}
```

**Key rules:**
- The variant name matches the Rust enum variant exactly (PascalCase, e.g., `Phonepe`, `Stripe`, `Adyen`)
- `Secret<String>` fields are **plain strings** (NOT `{"value":"..."}`)
- The outer key is always `"config"`

**Find the correct variant name:** Check the generated proto code or `backend/domain_types/src/router_data.rs` for the `ConnectorSpecificConfig` enum variant.

**Example (PhonePe):**
```
x-connector-config: {"config":{"Phonepe":{"merchant_id":"JUSPAYUAT","salt_key":"<salt>","salt_index":"1"}}}
```

### Step 3: Construct the Request Body

The body uses **protobuf JSON encoding** (NOT serde). Key difference:

- `SecretString` proto message has `message SecretString { string value = 1; }`, so in protobuf JSON it's `{"value":"..."}`
- Enums are string names
- Standard protobuf JSON rules apply

**Example (TriggerOtpForWallet):**
```json
{
  "mobile_number": {"value": "8178392166"},
  "merchant_id": {"value": "JUSPAYUAT"}
}
```

### Step 4: Send the gRPC Request

```bash
grpcurl -plaintext \
  -H 'x-connector-config: <JSON from Step 2>' \
  -H 'x-merchant-id: <merchant_id>' \
  -H 'x-tenant-id: default' \
  -H 'x-request-id: test_001' \
  -H 'x-connector-request-reference-id: test_ref_001' \
  -d '<JSON from Step 3>' \
  localhost:8000 types.PaymentService/<RpcMethod>
```

**Required headers:**
| Header | Value |
|--------|-------|
| `x-connector-config` | Connector-specific config JSON (Step 2) |
| `x-merchant-id` | Merchant identifier |
| `x-tenant-id` | Tenant identifier (use `default` for testing) |
| `x-request-id` | Unique request ID for tracing |
| `x-connector-request-reference-id` | Reference ID for connector correlation |

**Find the RPC method name:** Check `backend/grpc-api-types/proto/services.proto` for the service and method name.

### Step 5: Validate the Response

The response should contain:

1. **`rawConnectorRequest`**: Verify the outgoing URL, headers, and body match the Haskell implementation
2. **`rawConnectorResponse`**: The raw response from the connector
3. **Business fields**: Extracted response data (e.g., `otpToken`, `connectorTransactionId`)
4. **`statusCode`**: HTTP status from the connector

**Validation checklist:**
- [ ] URL in `rawConnectorRequest` points to the correct connector endpoint
- [ ] Request headers match the Haskell implementation (e.g., `X-VERIFY`, `Authorization`)
- [ ] Request body matches the Haskell implementation (decode Base64 payloads if applicable)
- [ ] Connector returned a well-formed response (success or a recognized business error)
- [ ] Response fields are correctly extracted into the gRPC response message
- [ ] If connector returns success, the business data is correct

**Success criteria (STRICT — iterate until success):**

A test is only complete when the connector returns HTTP 200 with correct business data.
Do NOT stop on failure. Debug, fix code, rebuild, retest. Repeat until HTTP 200.

- **PASS:** HTTP 200 with correct business data from the connector = task succeeds.
  This is the ONLY acceptable outcome. Debug, fix, rebuild, and retest until you
  get here.

- **CAPABILITY MISMATCH (switch connector):**
  The connector returns "feature not enabled", "recurring not supported", "not
  available in test mode", or similar errors indicating the connector's test
  environment cannot support this flow. This is NOT a code bug — it is an
  environment limitation.
  Action: switch to the next connector from the analysis report's connector list.
  Implement the new connector's integration, execute the full pre-call chain,
  and retest. Continue until a working connector is found.

- **FAIL (must debug, fix, rebuild, and retry):**
  - "Api Mapping Not Found" or similar routing error → **WRONG URL**. Go to
    euler-api-txns (the gateway's Endpoints.hs / Flow.hs), find the correct URL,
    fix the endpoint or base_url in your implementation, rebuild, and retest.
  - Invalid merchant / credential errors → wrong config format, wrong variant name,
    or bad credentials. Fix and retest.
  - Connection error or parse failure → code bug. Fix and retest.
  - gRPC error (proto mismatch, unknown service) → registration bug. Fix and retest.
  - **ANY response indicating the request never reached the connector's actual
    business logic is a FAIL.**
  - **ANY non-200 where pre-calls were available but skipped is a FAIL.**
  - **ANY non-200 where dummy/placeholder values were used instead of real
    pre-call outputs is a FAIL.** "I used a dummy token" is never valid.

- **LAST RESORT (only after exhausting ALL connectors):**
  If every connector from the analysis report has been tried and ALL return
  environment-level capability limitations (not code bugs), escalate to user with:
  (a) Every connector tried and the specific error from each
  (b) Evidence that pre-call chain was fully implemented and executed for each
  (c) Evidence that errors are from the connector's business logic, not our code
  (d) Verification that URL, headers, body match Haskell for each connector

**Do NOT proceed to commit/PR with a non-200 result unless LAST RESORT applies.**
**Do NOT accept "I used a dummy token" as a reason for non-200 — implement and
execute the pre-call that produces the real token.**

**URL verification step:** Before sending ANY test request, cross-reference the URL
your implementation uses against the Haskell source:
1. Read the gateway's Endpoints.hs for all endpoint URLs
2. Read the gateway's Flow.hs to see which URL this specific flow uses
3. Verify the base URL domain is correct for the API category (e.g., PG vs Wallet)
4. If `config/development.toml` needs a temporary URL change for testing, note the
   original value — you must revert it before committing

**Note:** Sandbox/preprod environments may return synthetic values (e.g., PhonePe sandbox returns `merchantId: "MERCHANT123"` and `otpToken: "OTP6789"` instead of real values). This is expected connector sandbox behavior, not a bug.

### Step 6: Verify Checksums/Signatures Independently

For connectors using request signing (e.g., PhonePe X-VERIFY, Razorpay signature):

```bash
# Example: Verify PhonePe X-VERIFY checksum
BASE64='<base64 payload from rawConnectorRequest>'
ENDPOINT='/v3/merchant/otp/send'
SALT='<salt_key>'
HASH=$(printf "%s%s%s" "$BASE64" "$ENDPOINT" "$SALT" | shasum -a 256 | cut -d' ' -f1)
echo "Computed: ${HASH}###<salt_index>"
# Compare with X-VERIFY header in rawConnectorRequest
```

### Common Issues

| Symptom | Cause | Fix |
|---------|-------|-----|
| `Failed to parse X-Connector-Config JSON` | Wrong JSON format for header | Use serde format: `{"config":{"VariantName":{...}}}` with plain strings for `Secret<String>` |
| `X-Connector-Config header missing config` | Missing `config` key or wrong variant name | Check exact enum variant name in generated code |
| `InvalidConnectorConfig: secondary_base_url` | Flow needs `secondary_base_url` but not configured | Add `<connector>.secondary_base_url` to `config/development.toml` |
| `Api Mapping Not Found` (PhonePe) | Wrong base URL domain for the API category | PhonePe has separate domains for PG (Hermes) vs Wallet (Mercury/pg-sandbox). Use `secondary_base_url` for wallet flows |
| Connection refused on localhost:8000 | Server not started or crashed | Check `/tmp/grpc-server.log` for errors; ensure ports 8000 and 8080 are free |
| "Feature not enabled" / "Recurring not supported" | Connector doesn't support this feature in test mode | Capability mismatch — switch to fallback connector per analysis report. Not a code bug. |
| 4xx with "invalid token" / "entity not found" when using dummy values | Pre-call dependencies were skipped | Implement and execute the pre-call chain first. Do NOT use dummy values — get real outputs from prior flows. |

### PhonePe-Specific: Dual Base URL Architecture

PhonePe uses **two separate API gateways**:

| Gateway | Preprod Domain | Production Domain | APIs |
|---------|---------------|-------------------|------|
| **Hermes** (PG) | `api-preprod.phonepe.com/apis/hermes/` | `api.phonepe.com/apis/hermes/` | Pay, status, refund, VPA validate |
| **Mercury/pg-sandbox** (Wallet) | `api-preprod.phonepe.com/apis/pg-sandbox/` | `mercury-t2.phonepe.com/` | OTP send/verify, wallet balance/debit/topup, delink |

Configure in TOML:
```toml
phonepe.base_url = "https://api-preprod.phonepe.com/apis/hermes/"           # PG flows
phonepe.secondary_base_url = "https://api-preprod.phonepe.com/apis/pg-sandbox/" # Wallet flows
```

Wallet flow implementations must use `secondary_base_url` in their `get_url()`:
```rust
fn get_url(&self, req: &RouterDataV2<...>) -> CustomResult<String, errors::ConnectorError> {
    let secondary_base_url = req
        .resource_common_data
        .connectors
        .phonepe
        .secondary_base_url
        .as_ref()
        .ok_or(errors::ConnectorError::InvalidConnectorConfig {
            config: "secondary_base_url",
        })?;
    Ok(format!("{}{}", secondary_base_url, constants::API_OTP_SEND_ENDPOINT))
}
```

## 8.1.6 PR Creation (After Successful gRPC Test)

Once the gRPC smoke test passes (HTTP 200), raise a pull request.
This is a mandatory step — do not skip it or leave it for later.

### Pre-PR Steps

**1. Revert any test-only config changes:**

If you modified `config/development.toml` (or other config files) solely for testing
purposes (e.g., changed a base_url to point to a different preprod environment),
revert those changes before committing. The committed code must have the correct
production-ready config values.

```bash
git diff config/  # check what changed
# If any changes are test-only, revert them:
git checkout config/development.toml
```

**2. Run pre-commit checks:**

```bash
# Both must pass before committing
cargo clippy --all-targets --all-features
cargo +nightly fmt --all
```

### Commit and Push

```bash
git add -A
git commit -m "feat(<connector>): add <FlowName> flow for <Connector>"
git push -u origin <branch-name>
```

Do NOT amend and force push unless explicitly asked. Use regular commits.

### PR Title Format

```
feat(<connector>): add <FlowName> flow for <Connector> <short description>
```

Examples:
- `feat(phonepe): add TriggerOtpForWallet flow for PhonePe wallet OTP`
- `feat(razorpay): add Authorize and PSync flows for Razorpay UPI`

### PR Description Template

The description must include:

1. **Summary** — brief bullet points of what was added (flows, layers touched, config changes,
   any pre-call dependency flows implemented on the same branch)
2. **How did you test this?** — the gRPC smoke test request and response (HTTP 200)
3. **Pre-call chain** — if pre-call dependencies were executed, document the chain
4. **If test-only config was used** — mention what was changed for testing and that it
   was reverted before commit

### Testing Section: What to Include and Exclude

**SECURITY: Strip all connector credentials from the PR description.**

The `x-connector-config` header contains connector secrets (API keys, salt keys,
merchant IDs, etc.) and MUST NOT appear in the PR. Show only the non-sensitive
parts of the test.

**Include in the PR:**

```markdown
## How did you test this?

### gRPC smoke test against <Connector> preprod — PASS

**Pre-call chain:**
<List each pre-call step executed, with the flow name and key output captured.
Or "No pre-calls needed" if the flow has no dependencies.>

**Request:**
\```bash
grpcurl -plaintext \
  -H 'x-connector-config: <REDACTED>' \
  -H 'x-merchant-id: <merchant_id>' \
  -H 'x-tenant-id: default' \
  -H 'x-request-id: test_001' \
  -H 'x-connector-request-reference-id: test_ref_001' \
  -d '<request body — replace only secrets (API keys, salt keys) with placeholders>' \
  localhost:8000 types.PaymentService/<FlowName>
\```

**Response (HTTP 200 — SUCCESS):**
\```json
<FULL response JSON exactly as returned — do NOT redact any part of the response>
\```
```

**Include if test-only config was used:**

```markdown
**Note:** During testing, `config/development.toml` had `<connector>.<field>` set
to `<test value>` to reach the <environment> endpoint. This has been reverted to
the standard value before commit. To reproduce the test locally, set this URL in
your development config.
```

**Rules for sanitizing:**
- **`x-connector-config` header:** Replace the entire value with `<REDACTED>`.
  This header contains connector secrets (API keys, salt keys) and must never appear
  in plain text in the PR.
- **Request body (`-d` argument):** Replace only secrets (API keys, salt keys, auth
  tokens) with `<PLACEHOLDER>` descriptors. Non-secret identifiers like merchant_id
  and phone numbers can stay as-is.
- **Response body:** Include the FULL response JSON **exactly as returned** by the
  connector. Do NOT redact or placeholder any part of the response — reviewers need
  the complete raw output (rawConnectorRequest, rawConnectorResponse, checksums,
  base64 payloads, field values, headers, everything) to verify correctness.

### If Connector Credentials Are Needed

If you need preprod/sandbox credentials for a connector and don't have them,
**ask the user before proceeding**. Do not guess or use placeholder credentials
for the actual test — the test must hit the real preprod endpoint.
