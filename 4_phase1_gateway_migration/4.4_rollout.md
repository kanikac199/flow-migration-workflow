## Rollout Stages

```
Stage 0: Internal Testing      (0% real traffic)
    |
Stage 1: Shadow Mode           (0% real traffic, both paths called)
    |
Stage 2: Canary 1%             (1% real traffic via feature flag)
    |
Stage 3: Canary 10%            (10% real traffic)
    |
Stage 4: Partial Rollout 50%   (50% real traffic)
    |
Stage 5: Full Rollout 100%     (100% real traffic, old path as fallback)
    |
Stage 6: Cleanup               (remove old path code after 2-week soak)
```

---

## Stage 0: Internal Testing

**Duration:** 1-3 days

- [ ] Deploy connector-service with the new connector to staging
- [ ] Deploy euler_txn with the bridge code + feature flag (disabled)
- [ ] Enable feature flag for internal test merchant IDs only
- [ ] Run manual test transactions through the new path:
  - Successful payment
  - Failed payment
  - Timeout scenario
  - Invalid card / payment method
  - Partial capture (if applicable)
  - Void (if applicable)
  - Refund
- [ ] Verify all responses match expected behavior

**Go/No-Go:** All manual tests pass. No unexpected errors in logs.

---

## Stage 1: Shadow Mode

**Duration:** 1 week

In shadow mode, euler_txn calls **both** the old path and the new path for every transaction. The old path's response is used as the actual response. The new path's response is logged for comparison.

```
Real Request --> euler_txn
                    |
                    +--> Old Path (gateway adapter) --> Response A (USED)
                    |
                    +--> New Path (connector-service) --> Response B (LOGGED)
                    |
                    +--> Compare A vs B --> Log discrepancies
```

**Implementation:** Add shadow call in the feature flag block:

```haskell
if shadowModeEnabled
  then do
    -- Call old path (actual response)
    responseA <- oldPathGatewayCall ...
    -- Call new path (shadow, async/fire-and-forget)
    _ <- async $ do
      responseB <- ConnectorService.Flow.makeConnectorServiceRequest ...
      compareAndLogDiscrepancy responseA responseB gateway merchantId
    pure responseA
  else ...
```

**Monitoring:** Compare these fields:
- Transaction status mapping
- Gateway transaction ID
- Error code / error message
- Redirect URL (if applicable)
- Response latency

**Go/No-Go:** Zero discrepancies in status mapping for 7 consecutive days.

---

## Stage 2: Canary 1%

**Duration:** 1 week

- [ ] Set traffic percentage: `CONNECTOR_SERVICE_<GATEWAY>_TRAFFIC_PCT = 1`
- [ ] Monitor continuously:

| Metric | Threshold | Action if Breached |
|--------|-----------|-------------------|
| Error rate delta | > 0.5% vs old path | Rollback immediately |
| p99 latency delta | > 200ms vs old path | Investigate; rollback if persistent |
| Success rate delta | > 0.1% drop vs old path | Rollback immediately |
| Status mismatch | Any single instance | Investigate immediately |

**Go/No-Go:** All metrics within thresholds for 7 consecutive days.

---

## Stage 3: Canary 10%

**Duration:** 1 week

- [ ] Set traffic percentage: `CONNECTOR_SERVICE_<GATEWAY>_TRAFFIC_PCT = 10`
- [ ] Same monitoring thresholds as Stage 2
- [ ] At this scale, edge cases will surface:
  - International cards
  - Multi-currency transactions
  - High-value transactions
  - Timeout/retry scenarios

**Go/No-Go:** All metrics within thresholds for 7 consecutive days. No new edge case failures.

---

## Stage 4: Partial Rollout 50%

**Duration:** 1 week

- [ ] Set traffic percentage: `CONNECTOR_SERVICE_<GATEWAY>_TRAFFIC_PCT = 50`
- [ ] Same monitoring thresholds
- [ ] Review: total transaction volume through new path is significant

**Go/No-Go:** All metrics within thresholds for 7 consecutive days.

---

## Stage 5: Full Rollout 100%

**Duration:** 2+ weeks (soak period)

- [ ] Set traffic percentage: `CONNECTOR_SERVICE_<GATEWAY>_TRAFFIC_PCT = 100`
- [ ] Old path code remains in codebase as fallback
- [ ] Feature flag can still be flipped to 0% for instant rollback
- [ ] Monitor for full 2 weeks at 100%

**Go/No-Go:** All metrics stable for 2 consecutive weeks. No rollbacks needed.

---

## Stage 6: Cleanup

- [ ] Remove the old gateway adapter code from euler_txn's `Gateway/<Name>/Flow.hs`
  - Or: mark as deprecated with a comment + date
- [ ] Remove the shadow mode code path
- [ ] Simplify the feature flag (can remove the traffic percentage logic)
- [ ] Update documentation
- [ ] Close the migration tracking ticket/issue

---

## Rollback at Any Stage

At **any** stage, rollback is:

1. Set `CONNECTOR_SERVICE_<GATEWAY>_TRAFFIC_PCT = 0` (or disable the feature flag)
2. Verify traffic flows through old path (check metrics dashboard)
3. Investigate root cause
4. Fix and restart from the appropriate stage

See: [8.3_rollback_plan.md](../8_testing_and_operations/8.3_rollback_plan.md) for detailed rollback procedures.
