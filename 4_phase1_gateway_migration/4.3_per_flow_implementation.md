## Prerequisites

Before implementing a flow, check whether the **flow type** already exists in connector-service.
All flows in the table below (Authorize, PSync, Capture, Void, Refund, RSync, etc.) already have
flow markers, sub-traits, proto RPCs, and gRPC handlers in the codebase.

**If the flow type you need does NOT exist** (e.g., TriggerOtp, VerifyOtp, CheckBalance, or any
wallet/OTP/custom flow that isn't in the table below), you must create the full flow infrastructure
first. Follow **[4.5_new_flow_type.md](4.5_new_flow_type.md)** before returning here.

## Implementation Order Per Gateway

Follow this order for each gateway. Authorize and PSync are the minimum viable migration.

```
Step 1: Authorize  (required -- the core payment flow)
    |
Step 2: PSync      (required -- transaction status sync)
    |
Step 3: Refund     (required for production)
    |
Step 4: RSync      (required -- refund status sync)
    |
Step 5: Capture    (if gateway supports pre-auth)
    |
Step 6: Void       (if gateway supports pre-auth)
    |
Step 7: SetupMandate / RepeatPayment  (if gateway supports mandates)
    |
Step 8: Remaining flows (IncrementalAuth, MandateRevoke, etc.)
```

## Per-Flow Implementation Notes

### Authorize

**euler_txn source:** `Gateway/<Name>/Flow.hs` -- `initiateTxn` function

Key considerations:
- euler_txn may have V1/V2 flow variants (e.g., HDFC V1 TripleDES vs V2 AES)
- The response may include a redirect URL (for 3DS / bank redirect flows)
- Pre-auth (CaptureMethod::Manual) vs auto-capture (CaptureMethod::Automatic) affects the request
- Zero-auth (amount=0) may require special handling

### PSync (Payment Status Sync)

**euler_txn source:** `Gateway/<Name>/Flow.hs` -- `syncTxnStatus` or `initSyncGatewayStatus`

Key considerations:
- This is called by the TxnSyncWorkFlow (ProcessTracker) repeatedly until terminal state
- The request typically includes the gateway transaction ID (`epgTxnId`)
- The response must map to the exact same `TxnStatus` that euler_txn's old path would produce
- Some gateways use polling, others use reference ID lookup

### Capture

**euler_txn source:** `Gateway/<Name>/Flow.hs` -- `processCapture`

Key considerations:
- Only applicable for pre-auth gateways (HDFC, Cybersource, Stripe, etc.)
- May support partial capture
- Amount in capture may differ from authorization amount

### Void

**euler_txn source:** `Gateway/<Name>/Flow.hs` -- `processVoid`

Key considerations:
- Voids a pre-authorized transaction before capture
- Some gateways don't support void (use refund instead)
- May support partial void

### Refund

**euler_txn source:** `Gateway/<Name>/Transforms.hs` -- `make_<Gw>RefundRequest`

Key considerations:
- May be order-based (by order ID) or transaction-based (by txn ID)
- Partial refunds must be supported if the gateway allows them
- The refund ID/reference must be preserved for sync

### RSync (Refund Status Sync)

**euler_txn source:** `Gateway/<Name>/Flow.hs` -- `syncRefundStatus`

Key considerations:
- Similar to PSync but for refund transactions
- Called by the REFUND_WORKFLOW ProcessTracker

## Gateway-Specific Patterns to Watch For

| Pattern | Affected Gateways | How to Handle |
|---------|------------------|---------------|
| **3DS enrollment** | HDFC, Cybersource, MPGS | Multi-step: enrollment -> redirect -> auth. Use composite-service or handle in bridge. |
| **Request encryption** | HDFC (AES/TripleDES), some others | Implement encryption in connector-service Rust code |
| **XML requests/responses** | Some legacy gateways | Use `quick-xml` in connector-service transformers |
| **Form-encoded requests** | PayU, CCAvenue, many redirect gateways | Use `RequestContent::FormUrlEncoded` |
| **Signature-based auth** | Billdesk, Razorpay, Cybersource | Implement HMAC/RSA signing in connector |
| **Certificate auth (mTLS)** | Some enterprise gateways | Use `get_certificate()` method in ConnectorIntegrationV2 |
| **Multi-currency** | International gateways (Stripe, Adyen) | Ensure currency code mapping is correct |
| **Token-based payments** | Most gateways via stored cards | Token data flows through `payment_method_data` |
