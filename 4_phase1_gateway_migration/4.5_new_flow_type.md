# 4.5 Adding a New Flow Type to connector-service

> **Called by:** [1_orchestrator.md](../1_orchestrator.md) — when the euler_txn flow has no existing equivalent in connector-service
> **Prerequisite:** [4.1_create_connector.md](4.1_create_connector.md) completed (connector struct exists)
> **Reference:** [7.3_file_references.md](../7_reference/7.3_file_references.md)

---

## When to Use This Document

Use this guide when migrating an euler_txn flow that does NOT have a corresponding flow marker type in connector-service. For example:

- `triggerOTPForWallet` — no `TriggerOtpForWallet` marker exists
- `verifyVpa` — no `VerifyVpa` marker exists
- `checkBalance` — no `CheckBalance` marker exists

If the flow marker already exists (e.g., `Authorize`, `Refund`, `PSync`), skip this document and go directly to [4.3_per_flow_implementation.md](4.3_per_flow_implementation.md).

---

## Overview: What You Must Create

Adding a new flow type requires changes across **7 layers** of the connector-service architecture:

```
Layer 1: Flow marker type          (connector_flow.rs)
Layer 2: Domain request/response   (connector_types.rs in domain_types)
Layer 3: Sub-trait definition      (connector_types.rs in interfaces)
Layer 4: Proto messages + RPC      (payment.proto + services.proto)
Layer 5: Proto-to-domain conversions (types.rs in domain_types)
Layer 6: gRPC handler              (server/*.rs in grpc-server)
Layer 7: Default implementations   (default_implementations.rs)
```

Then implement the actual connector flow (mod.rs + transformers.rs) per [4.1_create_connector.md](4.1_create_connector.md).

---

## Layer 1: Flow Marker Type

**File:** `backend/domain_types/src/connector_flow.rs`

Add a zero-sized marker struct and a `FlowName` enum variant:

```rust
// Add with the other flow marker structs
pub struct <NewFlowName>;    // e.g., pub struct TriggerOtpForWallet;

// Add to the FlowName enum
pub enum FlowName {
    // ... existing variants ...
    <NewFlowName>,           // e.g., TriggerOtpForWallet,
}
```

Follow naming conventions:
- Struct: PascalCase, descriptive (e.g., `TriggerOtpForWallet`, `VerifyVpa`)
- FlowName variant: same name as the struct

---

## Layer 2: Domain Request/Response Types

**File:** `backend/domain_types/src/connector_types.rs`

Define the flow-specific request and response data structs:

```rust
/// Request data for the new flow
#[derive(Debug, Clone)]
pub struct <NewFlow>Data {
    // Fields needed to make the gateway request
    // Map from the euler_txn function parameters
    pub field1: Type1,
    pub field2: Secret<String>,  // Use Secret for sensitive data
    // ...
}

/// Response data from the new flow
#[derive(Debug, Clone)]
pub struct <NewFlow>ResponseData {
    // Fields returned by the gateway
    // Map from the euler_txn response type
    pub success: bool,
    pub code: String,
    pub message: String,
    pub status_code: u16,
    // ... flow-specific response fields
}
```

**Guidelines:**
- Only include fields that the connector needs. Orchestration/caching (like Redis OTP token storage) stays in euler_txn.
- Use `Secret<String>` for sensitive data (API keys, tokens, PII).
- Use `Option<T>` for fields that may be absent in responses.

### Decision: Reuse `PaymentsResponseData` or Create a New Struct?

**Do NOT create a new response data type by default.** First check whether the flow's
response can fit into an existing type. The rule is based on **semantic domain**, not
field count:

**Reuse `PaymentsResponseData`** (add a variant if needed) when:
- The flow is semantically a **payment transaction operation** — authorization,
  sync, capture, void, 3DS authentication, incremental auth, etc.
- The response carries payment-transaction concepts: `resource_id`, `redirection_data`,
  `connector_metadata`, `mandate_reference`, `network_txn_id`, `authentication_data`.
- Example: 5 of 7 `PaymentsResponseData` variants are used by only ONE flow each
  (SdkSessionToken, PreAuthenticate, Authenticate, PostAuthenticate,
  IncrementalAuthorization). Single-flow usage is fine — what matters is the domain.

**Create a standalone response struct** when:
- The flow is a **fundamentally different operation** — wallet balance check, OTP
  trigger, mandate lifecycle, order creation, VPA verification, etc.
- The response fields have **zero semantic overlap** with payment transaction concepts
  (e.g., `available_balance`, `otp_token`, `mandate_status`).
- The flow maps to a **different gRPC service** (e.g., `RecurringPaymentService`
  instead of `PaymentService`).

**Similarly, reuse existing shared types** for other response categories:
- `RefundsResponseData` — for refund-related flows (Refund, RSync)
- `DisputeResponseData` — for dispute-related flows (Accept, SubmitEvidence, DefendDispute)

**Reference — current response type reuse in connector-service:**

| Response Type | # Flows | Domain |
|---|---|---|
| `PaymentsResponseData` (7 variants) | 12 | Payment transactions |
| `RefundsResponseData` | 2 | Refunds |
| `DisputeResponseData` | 3 | Disputes |
| `PaymentCreateOrderResponse` | 1 | Pre-auth order creation |
| `MandateRevokeResponseData` | 1 | Mandate lifecycle |
| `TriggerOtpForWalletResponseData` | 1 | Wallet OTP |
| `RefreshWalletBalanceResponseData` | 1 | Wallet balance |

**The same logic applies to request data types.** `PaymentsAuthorizeData` has 40+ fields
for a full payment authorization — do NOT reuse it for a 2-field OTP trigger. Create a
dedicated request struct when the flow's inputs are fundamentally different from existing
request types.

## Layer 3: Sub-Trait Definition

**File:** `backend/interfaces/src/connector_types.rs`

### Step 3a: Define the new sub-trait

```rust
use domain_types::connector_flow::<NewFlowName>;
use domain_types::connector_types::{<NewFlow>Data, <NewFlow>ResponseData};

// Add the sub-trait definition (follow existing patterns like PaymentAuthorizeV2, RefundV2)
pub trait <NewFlow>V2<T: PaymentMethodDataTypes = DefaultPaymentMethodData>:
    ConnectorIntegrationV2<
        <NewFlowName>,
        PaymentFlowData,     // or a custom ResourceCommonData if needed
        <NewFlow>Data,
        <NewFlow>ResponseData,
    >
{
}
```

**Note on generic type parameter `T`:** If the flow does NOT involve payment method data (e.g., OTP flows, VPA verification), you can omit `T` and use a non-generic trait:

```rust
pub trait <NewFlow>V2:
    ConnectorIntegrationV2<
        <NewFlowName>,
        PaymentFlowData,
        <NewFlow>Data,
        <NewFlow>ResponseData,
    >
{
}
```

### Step 3b: Add to ConnectorServiceTrait

```rust
pub trait ConnectorServiceTrait<T: PaymentMethodDataTypes>:
    ConnectorCommon
    + ValidationTrait
    + PaymentAuthorizeV2<T>
    // ... existing sub-traits ...
    + <NewFlow>V2              // <-- add your new sub-trait here
    + VerifyRedirectResponse
{
}
```

## Layer 4: Proto Messages and RPC

> **IMPORTANT: Naming Conventions**
>
> All field names, type names, and enum variants MUST follow **connector-service**
> conventions, NOT euler-api-txns (Haskell/PureScript) conventions. The Haskell
> codebase is the **source of business logic**, not the source of naming.
>
> Before adding ANY new proto field or Rust struct field:
> 1. Grep existing `.proto` files in `crates/types-traits/grpc-api-types/proto/`
>    for the concept (e.g., phone, address, amount, status, token, customer, etc.)
> 2. Grep existing Rust domain types in `crates/types-traits/domain_types/src/`
> 3. Use whatever name the codebase already uses for that concept
> 4. Only invent a new name if the concept is genuinely new to connector-service
>
> Gateway-specific serialization (e.g., `#[serde(rename = "mobileNumber")]`) is fine —
> that matches the external API contract. Internal proto/domain fields must follow
> connector-service naming.

### Step 4a: Proto messages

**File:** `backend/grpc-api-types/proto/payment.proto`

Add request and response messages:

```proto
// Request message for the new flow
message <NewFlow>ServiceRequest {
    string field1 = 1;
    string field2 = 2;
    // ... map from domain request type fields
    // Follow existing naming conventions (snake_case for proto fields)
}

// Response message for the new flow
message <NewFlow>ServiceResponse {
    bool success = 1;
    string code = 2;
    string message = 3;
    // ... map from domain response type fields
    optional ErrorInfo error = 10;
    int32 status = 11;
}
```

### Step 4b: Service definition

**File:** `backend/grpc-api-types/proto/services.proto`

Either add an RPC to an existing service or create a new service:

**Option A: Add to existing service** (if the flow is a variant of an existing category):
```proto
service PaymentService {
    // ... existing RPCs ...
    rpc <NewFlowName> (<NewFlow>ServiceRequest) returns (<NewFlow>ServiceResponse);
}
```

**Option B: Create new service** (if the flow is in a new category like wallets):
```proto
service <NewCategory>Service {
    rpc <NewFlowName> (<NewFlow>ServiceRequest) returns (<NewFlow>ServiceResponse);
}
```

### Step 5b: Domain → Proto (response generation)

```rust
pub fn generate_<new_flow>_response(
    router_data_v2: RouterDataV2<
        <NewFlowName>,
        PaymentFlowData,
        <NewFlow>Data,
        <NewFlow>ResponseData,
    >,
) -> Result<<NewFlow>ServiceResponse, error_stack::Report<ApplicationErrorResponse>> {
    match router_data_v2.response {
        Ok(response_data) => {
            Ok(<NewFlow>ServiceResponse {
                success: response_data.success,
                code: response_data.code,
                message: response_data.message,
                // ... map domain response fields to proto fields
                error: None,
                status: router_data_v2.resource_common_data.status as i32,
            })
        }
        Err(err) => {
            Ok(<NewFlow>ServiceResponse {
                success: false,
                code: err.code.clone(),
                message: err.message.clone(),
                error: Some(grpc_api_types::payments::ErrorInfo {
                    code: err.code,
                    message: err.message,
                    // ...
                }),
                status: common_enums::AttemptStatus::Failure as i32,
                // ...
            })
        }
    }
}
```

## Layer 6: gRPC Handler

### Step 6a: Implement the handler

**If you added an RPC to an existing service** (e.g., PaymentService):

Add the method to the existing `impl PaymentService for Payments` in `backend/grpc-server/src/server/payments.rs`.

**If you created a new service:**

Create a new file: `backend/grpc-server/src/server/<new_category>.rs`

**Pattern A: Using the `implement_connector_operation!` macro** (preferred for simple flows)

**File:** `backend/grpc-server/src/utils.rs`

```rust
implement_connector_operation!(
    fn_name: internal_<new_flow>,
    log_prefix: "<NEW_FLOW>",
    request_type: <NewFlow>ServiceRequest,           // proto request
    response_type: <NewFlow>ServiceResponse,          // proto response
    flow_marker: <NewFlowName>,                       // flow marker type
    resource_common_data_type: PaymentFlowData,       // domain common data
    request_data_type: <NewFlow>Data,                 // domain request data
    response_data_type: <NewFlow>ResponseData,        // domain response data
    request_data_constructor: <NewFlow>Data::foreign_try_from,          // proto->domain
    common_flow_data_constructor: PaymentFlowData::foreign_try_from,   // proto->domain
    generate_response_fn: generate_<new_flow>_response,                // domain->proto
    all_keys_required: None
);
```

Then in the service impl:

```rust
#[tonic::async_trait]
impl <NewCategory>Service for <NewCategory> {
    async fn <new_flow_name>(
        &self,
        request: tonic::Request<<NewFlow>ServiceRequest>,
    ) -> Result<tonic::Response<<NewFlow>ServiceResponse>, tonic::Status> {
        let config = get_config_from_request(&request)?;
        let service_name = "NewCategoryService".to_string();
        grpc_logging_wrapper(
            request,
            &service_name,
            config,
            FlowName::<NewFlowName>,
            |request_data| {
                Box::pin(async move {
                    let response = self.internal_<new_flow>(
                        &config,
                        request_data.payload,
                        request_data.extracted_metadata,
                        &request_data.masked_metadata,
                    ).await;
                    Ok(tonic::Response::new(response))
                })
            },
        ).await
    }
}
```

### Step 6b: Register the service on the server

**File:** `backend/grpc-server/src/app.rs`

Add the service to the `Service` struct:

```rust
pub struct Service {
    // ... existing services ...
    pub <new_category>_service: crate::server::<new_category>::<NewCategory>,
}
```

Add to the server builder in `grpc_server()`:

```rust
Server::builder()
    // ... existing .add_service() calls ...
    .add_service(
        <new_category>_service_server::<NewCategory>ServiceServer::new(
            self.<new_category>_service,
        ),
    )
```

Add the module to `backend/grpc-server/src/server.rs`:

```rust
pub mod <new_category>;
```

## Layer 7: Default Implementations

**File:** `backend/connector-integration/src/default_implementations.rs`

Create a default macro that provides an empty (NotImplemented) implementation for all connectors except the one(s) you're implementing:

```rust
// Define the macro
macro_rules! default_impl_<new_flow>_v2 {
    ($($connector:ident),*) => {
        $(
            impl<T: PaymentMethodDataTypes + std::fmt::Debug + Sync + Send + 'static + Serialize>
                connector_types::<NewFlow>V2<T> for $connector<T> {}

            impl<T: PaymentMethodDataTypes + std::fmt::Debug + Sync + Send + 'static + Serialize>
                ConnectorIntegrationV2<
                    <NewFlowName>,
                    PaymentFlowData,
                    <NewFlow>Data,
                    <NewFlow>ResponseData,
                > for $connector<T>
            {
                // Empty impl — returns NotImplemented error by default
            }
        )*
    };
}

// Invoke the macro for ALL connectors EXCEPT the one(s) implementing this flow
default_impl_<new_flow>_v2!(
    Razorpay,
    Paytm,
    // ... every other connector ...
    // Do NOT include the connector that implements this flow
);
```

**Study an existing macro** (e.g., `default_impl_verify_webhook_source_v2!` or `default_impl_authorize_v2!`) to match the exact pattern used in this codebase.

## Verification Checklist

After implementing all 7 layers:

- [ ] `cargo clippy --all-targets --all-features` passes with zero errors/warnings
- [ ] `cargo +nightly fmt --all` produces no changes
- [ ] `cargo build` succeeds (check for `ResponseRouterData` 2-param usage)
- [ ] Flow marker is in `connector_flow.rs` AND `FlowName` enum
- [ ] Sub-trait is added to `ConnectorServiceTrait` bounds
- [ ] Proto messages compile (`protoc` or `cargo build` triggers tonic codegen)
- [ ] Proto-to-domain conversions exist in `types.rs`
- [ ] gRPC handler is implemented and registered on the server
- [ ] Default impls exist for all non-implementing connectors
- [ ] The specific connector has a real `ConnectorIntegrationV2` impl
- [ ] Response transformer uses `ResponseRouterData<Response, Self>` (2 type params, NOT 5)
