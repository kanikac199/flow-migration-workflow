# 3.2 Pre-Migration Checklist

> **Next:** [3.3_decision_trees.md](3.3_decision_trees.md)

---

Complete this checklist **before** starting migration of any gateway or flow.

## 1. Analysis Phase

### 1.1 Identify euler_txn Source Files

- [ ] Gateway adapter: `euler-x/src-generated/Gateway/<GatewayName>/Flow.hs`
- [ ] Transforms: `euler-x/src-generated/Gateway/<GatewayName>/Transforms.hs`
- [ ] Types: `euler-x/src-generated/Gateway/<GatewayName>/Types.hs`
- [ ] OLTP logic: `oltp/src-generated/Product/OLTP/<FlowName>.hs`
- [ ] Workflow (if async): `src/Euler/API/Txns/WorkFlow/<WorkflowName>.hs`

### 1.2 Map All Flow Variants

- [ ] List every function exported from the gateway's `Flow.hs`
- [ ] Identify which `TxnObjectType` values this gateway handles
  - (ORDER_PAYMENT, MANDATE_REGISTER, MANDATE_PAYMENT, CAPTURE, VOID, etc.)
- [ ] Identify which `TxnFlowType` values are relevant
  - (CARD_TRANSACTION, NET_BANKING, INTENT, COLLECT, REDIRECT, EMI, etc.)
- [ ] Note any V1/V2 flow branching
  - Example: HDFC has V1 (TripleDES + 3DS enrollment) vs V2 (AES-256 encrypted redirect)
- [ ] Document the payment methods supported (Card, UPI, Net Banking, Wallet, etc.)

### 1.3 Catalog Feature Toggles

- [ ] Search for gateway name in `euler-x/src-generated/Config/Constants.hs`
- [ ] Search for gateway name in `euler-x/src-generated/Config/Options.hs`
- [ ] Document every feature flag that affects this gateway's behavior
- [ ] Classify each toggle:
  - **Must replicate** in connector-service (gateway-specific behavior)
  - **Pass as metadata** from euler_txn (merchant-specific)
  - **Keep in euler_txn** (business logic toggle)
  - **Can drop** (obsolete or already-default behavior)

### 1.4 Identify External Dependencies

- [ ] Does it call other Juspay services?
  - EC Locker (card vaulting)
  - Adapter Service
  - Dashboard Service
  - GSM Classifier
  - Smart Converter
  - Passetto (encryption key management)
- [ ] Does it use encryption?
  - JWE / JWT
  - TripleDES
  - AES-256
  - RSA
  - Certificate-based signing
- [ ] Does it have special proxy requirements?
  - Outbound proxy
  - Sync proxy
  - TLS certificate pinning
- [ ] Does it use Kafka for event streaming?

### 1.5 Document Request/Response Contract

- [ ] Capture a sample request to the gateway API (from production logs or sandbox)
- [ ] Capture a sample success response from the gateway API
- [ ] Capture a sample error/failure response from the gateway API
- [ ] Note the authentication mechanism:
  - API key (header / body)
  - Signature (HMAC / RSA)
  - Certificate (mTLS)
  - OAuth (access token flow)
- [ ] Note the content type: JSON, XML, form-encoded, multipart
- [ ] Note any special headers required

### 1.6 Pre-call Dependency Analysis

- [ ] Does this flow require inputs from a prior flow? (IDs, tokens, auth references)
  - List each dependency: flow name → what it produces → needed by target flow
- [ ] For each dependency, is it implemented in connector-service for this connector?
  - Real implementation / Empty stub / Flow type doesn't exist
- [ ] Is the connector's test/sandbox environment capable of supporting this flow?
  - Check for known limitations (production-only, bank approval required, etc.)
- [ ] If primary connector can't support testing, which fallback connector can?
- [ ] Document the full pre-call chain in execution order:
  ```
  Step 1: <Flow> → produces <output>
  Step 2: <Flow> → uses <input from step 1> → produces <output>
  Step 3: <Target Flow> → uses <inputs from steps 1-2>
  ```

## 2. Target State Design

### 2.1 Check Connector Service State

- [ ] Does the connector already exist in `backend/connector-integration/src/connectors/`?
  - If **YES**: List which flows are already implemented
  - If **NO**: Plan a new connector module
- [ ] Check `ConnectorEnum` in `backend/domain_types/src/connector_types.rs` for the variant
- [ ] Check `config/development.toml` for existing base URL configuration

### 2.2 Map Types

- [ ] Map euler_txn types to connector-service types using [7.1_type_mapping.md](../7_reference/7.1_type_mapping.md)
- [ ] Identify any types that need to be **added** to connector-service
- [ ] Identify any `AttemptStatus` mappings that are gateway-specific (non-standard)

### 2.3 Design Bridge Integration

- [ ] Review the existing bridge pattern in `euler-api-txns/euler-x/src-generated/Gateway/ConnectorService/`
- [ ] Determine if the existing `SDKParamsOrSendCollectReqBody` is sufficient or needs extension
- [ ] Determine if the existing `ConnectorServiceSdkResponse` covers all response fields needed
- [ ] Identify the euler_txn function where the bridge call will be inserted
  - (e.g., `initiateTxn` in `Gateway/<Name>/Flow.hs`)

---

## 3. Infrastructure Readiness

- [ ] **Feature flag configured**
  - Global toggle: `CONNECTOR_SERVICE_ENABLED` env var / ServiceConfiguration
  - Per-merchant toggle: merchant-level config in DB
  - Per-gateway toggle: gateway-level enablement
- [ ] **Monitoring dashboards** set up
  - Latency comparison (old path vs. new path)
  - Error rate comparison
  - Success rate comparison
  - Gateway-specific metrics
- [ ] **Alerting** configured
  - Threshold alerts for latency degradation (p99 > 200ms delta)
  - Threshold alerts for error rate increase (> 0.5% delta)
  - Threshold alerts for success rate drop (> 0.1% delta)
- [ ] **Rollback mechanism** verified
  - Feature flag can be flipped in < 1 minute
  - Rollback has been tested in staging
