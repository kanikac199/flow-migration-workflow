## Decision Tree 1: Should I Migrate This Gateway Now?

```
Is the gateway already in connector-service?
├── YES --> Does it support all flows needed by euler_txn?
│           ├── YES --> Proceed to bridge integration (4.2_build_bridge.md)
│           └── NO  --> Add missing flows first (4.1_create_connector.md), then bridge
│
└── NO  --> Is the gateway high-volume in euler_txn? (Tier 2)
            ├── YES --> Create new connector (4.1_create_connector.md), then bridge
            └── NO  --> Is there active maintenance burden in euler_txn?
                        ├── YES --> Migrate to reduce maintenance cost
                        └── NO  --> Defer; migrate opportunistically
```

## Decision Tree 2: How Should I Handle This Flow Variant?

```
Does connector-service have an equivalent flow type?
│
├── YES --> Use the existing flow
│          Examples:
│            euler_txn initiateTxn      --> connector-service Authorize
│            euler_txn syncTxnStatus    --> connector-service PSync
│            euler_txn processCapture   --> connector-service Capture
│            euler_txn processVoid      --> connector-service Void
│            euler_txn createRefund     --> connector-service Refund
│
└── NO  --> Is it a sub-variant of an existing flow?
            ├── YES --> Extend the existing flow with conditional logic
            │          Examples:
            │            Mandate register = Authorize + mandate_data in request
            │            Zero-auth = Authorize + amount=0
            │            Pre-auth = Authorize + CaptureMethod::Manual
            │            EMI = Authorize + emi_plan metadata
            │
            └── NO  --> Is it euler_txn-specific business logic?
                        ├── YES --> Keep in euler_txn; only migrate the gateway call
                        │          Examples:
                        │            Gateway decider logic
                        │            Split payment routing
                        │            Feature toggle evaluation
                        │            Merchant-specific validation
                        │
                        └── NO  --> Create a new flow type in connector-service
                                   (requires changes to domain_types/src/connector_flow.rs)
```

## Decision Tree 3: Where Should This Logic Live?

```
Is it gateway-specific?
│  (request/response format, auth headers, URL construction, error parsing)
│
├── YES --> connector-service (connector adapter)
│          Implementation in: connectors/<name>/mod.rs + transformers.rs
│
└── NO  --> Is it business logic?
            │  (validation, routing, state management, merchant rules)
            │
            ├── YES --> euler_txn (orchestration layer) -- for now
            │          Phase 2 may move this to a new orchestration service
            │          Examples:
            │            Order validation
            │            Gateway selection (decider)
            │            Transaction state machine updates
            │            Refund eligibility checks
            │            Mandate constraint validation
            │
            └── NO  --> Is it config/feature toggle logic?
                        │
                        ├── YES --> euler_txn (ServiceConfiguration / Redis)
                        │          Examples:
                        │            Merchant-level feature flags
                        │            Gateway scoring/routing weights
                        │            Rate limits
                        │
                        └── NO  --> Is it shared infrastructure?
                                    │
                                    ├── YES --> Evaluate:
                                    │          Encryption -> connector-service (if gateway-specific)
                                    │          Logging/metrics -> both services independently
                                    │          Proxy config -> both services independently
                                    │
                                    └── NO  --> Case-by-case evaluation
```

## Decision Tree 4: How to Handle Feature Toggles?

```
Is the toggle gateway-specific?
│  (e.g., "HDFC supports V2 flow", "PhonePe requires X header")
│
├── YES --> Replicate as connector-specific config in connector-service
│          Mechanism: ConnectorSpecificConfig in RouterDataV2
│          Or: hardcoded in the connector implementation
│
└── NO  --> Is the toggle merchant-specific?
            │  (e.g., "merchant X uses 3DS", "merchant Y has EMI enabled")
            │
            ├── YES --> Pass as metadata from euler_txn --> connector-service
            │          Mechanism: gRPC metadata headers or request fields
            │          euler_txn reads from DB/Redis, passes to connector-service
            │
            └── NO  --> Is the toggle global?
                        │  (e.g., "enable auto-retry", "shadow mode on")
                        │
                        ├── YES --> Keep in euler_txn's ServiceConfiguration
                        │          connector-service should not need global business toggles
                        │
                        └── NO  --> Is the toggle obsolete?
                                    │
                                    ├── YES --> Drop it during migration
                                    │          (document what was dropped and why)
                                    │
                                    └── NO  --> Evaluate; default to keeping in euler_txn
```
