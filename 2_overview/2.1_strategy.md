## What We Are Migrating

**Source system: `euler-api-txns`** (Haskell, transpiled from PureScript)

A monorepo handling the full payment transaction lifecycle:

- **53+ gateway adapters** in `euler-x/src-generated/Gateway/`
- **100+ API endpoints** defined in `src/Euler/API/Txns/Server.hs`
- **56 OLTP business logic modules** in `oltp/src-generated/Product/OLTP/`
- **18 async ProcessTracker workflows** in `src/Euler/API/Txns/WorkFlow/`
- **60+ database entity types** in `dbTypes/src-generated/EC/`

**Target system: `connector-service`** (Rust)

A stateless service with trait-based connector architecture:

- **75+ connector adapters** in `backend/connector-integration/src/connectors/`
- **gRPC + HTTP interface** via tonic + axum
- **Multi-language SDKs** (Java, Python, Node.js, Rust)
- **No database, no state** -- purely connector integration layer

---

## Migration Strategy: Phased Incremental via Bridge Pattern

An existing **bridge pattern** already works in production: `euler_txn` calls `connector-service` via HTTP for Razorpay UPI and PhonePe UPI flows. This is the proven path.

```
┌─────────────────┐         HTTP/gRPC          ┌─────────────────────┐
│   euler_txn     │ ──────────────────────────► │  connector-service  │
│  (orchestrator) │                             │  (connector layer)  │
│                 │ ◄────────────────────────── │                     │
│  - Auth/routing │      Unified response       │  - Build request    │
│  - DB state     │                             │  - Call gateway API  │
│  - Async flows  │                             │  - Parse response   │
│  - Webhooks     │                             │  - Error mapping    │
└─────────────────┘                             └─────────────────────┘
```

---

## Three Phases

| Phase | Scope | What Moves | What Stays in euler_txn | Risk |
|-------|-------|-----------|------------------------|------|
| **Phase 1** | Gateway adapters | Request building, gateway API calls, response parsing | Auth, DB, routing, async workflows, business logic | Low |
| **Phase 2** | Core transaction flows | Payment orchestration, refund routing, webhook parsing | DB state, ProcessTracker, merchant config | Medium |
| **Phase 3** | Async workflows | Transaction sync, refund sync, webhook delivery | *(minimal -- euler_txn becomes thin proxy)* | High |

**Phase 1 is the focus.** It is low-risk because euler_txn remains the orchestrator and owns all state. connector-service only handles the stateless gateway communication.

---

## Key Principles

1. **One gateway/flow at a time** -- never batch-migrate multiple gateways simultaneously
2. **Shadow traffic first** -- run both old and new paths, compare responses before switching
3. **Feature-flag everything** -- every migrated flow must be behind a merchant-level or global toggle
4. **Preserve the contract** -- euler_txn's API contracts to merchants must not change
5. **Stateless connector-service** -- connector-service must NOT take on DB/state responsibilities in Phase 1
6. **Exact behavioral parity** -- gateway requests, response status mappings, and error codes must match the old path exactly
7. **Rollback in seconds** -- flipping the feature flag must instantly revert to the old path

## Existing Bridge Reference (Production-Proven)

The bridge pattern is already live for two gateways:

| Gateway | Flow | euler_txn Bridge Code | Status |
|---------|------|----------------------|--------|
| Razorpay | UPI Intent/Collect | `euler-x/src-generated/Gateway/ConnectorService/Razorpay.hs` | Live |
| PhonePe | UPI Intent/Collect | `euler-x/src-generated/Gateway/ConnectorService/PhonePe.hs` | Live |

The bridge works as follows:

1. euler_txn builds a unified `SDKParamsOrSendCollectReqBody` request
2. euler_txn sends HTTP POST to connector-service with gateway auth in headers
3. connector-service builds the gateway-specific request, calls the gateway, parses the response
4. connector-service returns a unified `ConnectorServiceSdkResponse`
5. euler_txn maps the response back to its internal types (`PaymentResp`, `SdkParams`)

**Every new gateway migration follows this exact pattern.**
