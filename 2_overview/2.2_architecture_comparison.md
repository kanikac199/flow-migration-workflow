## Side-by-Side Comparison

| Aspect | euler_txn (Haskell) | connector-service (Rust) |
|--------|-------------------|------------------------|
| **Language** | Haskell (GHC 8.8.3), transpiled PureScript | Rust (1.70+) |
| **Framework** | euler-webservice (Servant + Warp) | tonic (gRPC) + axum (HTTP) |
| **Database** | MySQL (Beam ORM), Redis, PostgreSQL | **None** (stateless) |
| **Gateway dispatch** | `PgResponseReq` sum type (35+ constructors) | `ConnectorEnum` + trait dispatch via `convert_connector()` |
| **Request building** | Per-gateway `Transforms.hs` module | Per-connector `transformers.rs` module |
| **Response handling** | Gateway-specific parsers -> `TxnStatus` mapping | `handle_response_v2()` -> unified `PaymentsResponseData` |
| **Auth model** | API key in `MerchantAccount`, per-gateway auth in `MerchantGatewayAccount` | Auth passed via gRPC metadata headers (`x-auth`, `x-api-key`, etc.) |
| **Async workflows** | ProcessTracker scheduler (18 workflow types) | **None** (no scheduler) |
| **Config** | Env vars + DB `ServiceConfiguration` + Redis cache | TOML files + env vars |
| **Flow definition** | Monolithic handler functions in 16K-line modules | Zero-sized flow marker types + `ConnectorIntegrationV2` trait |
| **Boilerplate** | Manual per-gateway implementation | `macro_connector_implementation!` macro |
| **Multi-step flows** | Inline in handler code | `composite-service` crate orchestration |

## Architectural Layer Mapping

This shows which euler_txn layer maps to which connector-service layer:

```
euler_txn Layer                          connector-service Layer
─────────────────────────────────────    ─────────────────────────────────────
Server.hs (route registration)           grpc-server (gRPC service handlers)
    │                                        │
Helpers.hs (auth, decode, wrap)          gRPC metadata (auth headers)
    │                                        │
TranspiledTxns.hs (endpoint handlers)    composite-service (multi-step orchestration)
    │                                        │
OLTP/*.hs (business logic)               ── NOT IN SCOPE for connector-service ──
    │                                        │
Gateway/*/Flow.hs (gateway adapter)      connector-integration/connectors/*.rs
    │                                        │
Gateway/*/Transforms.hs (req/resp)       connectors/*/transformers.rs
    │                                        │
CommonGatewayTypes.hs (dispatch)         ConnectorEnum + convert_connector()
    │                                        │
HTTP call to gateway                     external-services (reqwest HTTP client)
```

**Key insight:** Only the bottom 4 layers (gateway adapter, transforms, dispatch, HTTP call) are in scope for Phase 1 migration. Everything above stays in euler_txn.

## Gateway Dispatch Pattern Comparison

### euler_txn: Sum Type Dispatch

```haskell
-- Gateway/CommonGatewayTypes.hs
data PgResponseReq
  = PAYU PayUPaymentResponse
  | RAZORPAY RazorpayRedirectResponse
  | PAYTM_V2 PayTMProcessTxnRespType
  | HDFC HDFCAuthorizeResponse
  | PHONEPE PhonePeFinalResponse
  | BILLDESK BilldeskResponse
  | CYBERSOURCE CybersourceResponse
  -- ... 35+ constructors
```

Each gateway response is a different type wrapped in a sum type constructor. Pattern matching dispatches to the correct handler.

### connector-service: Trait Dispatch

```rust
// connector-integration/src/types.rs
pub fn convert_connector(connector: ConnectorEnum) -> &'static dyn ConnectorServiceTrait<...> {
    match connector {
        ConnectorEnum::Stripe => &connectors::Stripe,
        ConnectorEnum::Razorpay => &connectors::Razorpay,
        ConnectorEnum::Phonepe => &connectors::Phonepe,
        // ... 75+ match arms
    }
}
```

Each connector is a zero-sized struct implementing the `ConnectorServiceTrait` super-trait. The runtime dispatches via trait object.

## Request/Response Pipeline Comparison

### euler_txn Pipeline

```
1. Read MerchantGatewayAccount from DB (auth credentials)
2. Build gateway-specific request (Gateway/<Name>/Transforms.hs)
3. Serialize (JSON/XML/form-encoded) + optional encrypt (AES/TripleDES/JWE)
4. HTTP call via euler-hs HTTP client (with proxy support)
5. Deserialize + optional decrypt
6. Parse into gateway-specific response type
7. Map gateway status -> TxnStatus
8. Update TxnDetail in DB
9. Return PaymentResp to caller
```

### connector-service Pipeline

```
1. Receive auth via gRPC metadata headers (x-auth, x-api-key, etc.)
2. Build gateway-specific request via TryFrom<RouterDataV2> (transformers.rs)
3. Serialize via serde (JSON/XML/form)
4. HTTP call via reqwest (with proxy support)
5. Deserialize response
6. Map to unified PaymentsResponseData via handle_response_v2()
7. Return via gRPC response
```

**The overlap is steps 2-6.** euler_txn's steps 1, 7-9 (DB reads/writes, status mapping to internal types) stay in euler_txn during Phase 1.

## Key Connector Service Abstractions

### RouterDataV2 (the core data carrier)

```rust
// backend/domain_types/src/router_data_v2.rs
pub struct RouterDataV2<Flow, ResourceCommonData, FlowSpecificRequest, FlowSpecificResponse> {
    pub flow: PhantomData<Flow>,
    pub resource_common_data: ResourceCommonData,
    pub connector_config: ConnectorSpecificConfig,
    pub request: FlowSpecificRequest,
    pub response: Result<FlowSpecificResponse, ErrorResponse>,
}
```

Generic over 4 type parameters:
- `Flow` -- zero-sized marker type (e.g., `Authorize`, `PSync`, `Refund`)
- `ResourceCommonData` -- shared data (e.g., `PaymentFlowData`, `RefundFlowData`)
- `FlowSpecificRequest` -- request payload (e.g., `PaymentsAuthorizeData`)
- `FlowSpecificResponse` -- response payload (e.g., `PaymentsResponseData`)

### ConnectorIntegrationV2 (the trait every connector implements per flow)

```rust
// backend/interfaces/src/connector_integration_v2.rs
pub trait ConnectorIntegrationV2<Flow, ResourceCommonData, Req, Resp> {
    fn get_headers(&self, req: &RouterDataV2<...>) -> Result<Vec<(String, Maskable<String>)>, ConnectorError>;
    fn get_content_type(&self) -> &'static str;
    fn get_http_method(&self) -> Method;
    fn get_url(&self, req: &RouterDataV2<...>) -> Result<String, ConnectorError>;
    fn get_request_body(&self, req: &RouterDataV2<...>) -> Result<Option<RequestContent>, ConnectorError>;
    fn build_request_v2(&self, req: &RouterDataV2<...>) -> Result<Option<Request>, ConnectorError>;
    fn handle_response_v2(&self, data: &RouterDataV2<...>, res: Response) -> Result<RouterDataV2<...>, ConnectorError>;
    fn get_error_response_v2(&self, res: Response) -> Result<ErrorResponse, ConnectorError>;
}
```

### ConnectorServiceTrait (the super-trait aggregating all flows)

```rust
// backend/interfaces/src/connector_types.rs
pub trait ConnectorServiceTrait<T>:
    ConnectorCommon
    + ValidationTrait
    + PaymentAuthorizeV2<T>
    + PaymentSyncV2
    + PaymentOrderCreate

    + PaymentAccessToken
    + CreateConnectorCustomer
    + PaymentTokenV2<T>
    + PaymentVoidV2
    + PaymentVoidPostCaptureV2
    + IncomingWebhook
    + RefundV2
    + PaymentCapture
    + SetupMandateV2<T>
    + RepeatPaymentV2<T>
    + AcceptDispute
    + RefundSyncV2
    + DisputeDefend
    + SubmitEvidenceV2
    + PaymentPreAuthenticateV2<T>
```

A connector must implement ALL sub-traits. For unsupported flows, use the `macro_connector_implementation!` macro to provide empty/default implementations.
